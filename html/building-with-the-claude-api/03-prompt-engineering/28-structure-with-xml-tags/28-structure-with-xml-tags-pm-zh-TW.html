使用 XML 標籤建立結構 — PM Perspective

一句話總結

XML 標籤是「把使用者資料安全塞進 prompt」的產品級做法——它讓 AI 繼續聽你的指令，而不是把使用者輸入誤解成新的命令。

PM 為什麼要關心 XML 標籤

每一個把使用者資料內插進 prompt 的 AI 功能都共享同一種失敗模式：使用者內容會和你的指令糊在一起，Claude 做出錯誤行為。XML 標籤是最省力、最高 leverage 的修法。

使用者內容類型 沒有標籤 有標籤
長文件 Claude 把文件當成指令的一部分 文件被關在 <document> 內作為資料分析
程式碼片段 Claude 把程式碼語法當成自然語言指令 程式碼關在 <my_code> 內作為資料 debug
多欄位的表單 欄位值混進散文 所有欄位被放進一個 <athlete_information> 容器
混合輸入（code + docs） Claude 無法區分 每種類型各自有標籤

這跟產品表單要加 label 是同一個道理：告訴使用者（或 AI）每一格欄位是做什麼用的。

心智模型：貼了標籤的檔案櫃抽屜

想像你把 20 頁銷售紀錄、5 頁公司政策和一個問題堆在同事桌上，一起遞過去。你大概會得到一句：「這是什麼？」

現在把同樣的材料放進一個有三個貼了標籤的抽屜的檔案櫃：銷售紀錄、公司政策、問題。你的同事會打開「問題」抽屜、讀完、然後只拉出他需要的那幾個抽屜。

XML 標籤就是抽屜上的標籤。<sales_records>...</sales_records> 在 prompt 裡就是一格抽屜。AI 知道要拉哪一格、哪一格可以先放著。

產品使用情境

XML 標籤最有用的場景

情境 為什麼標籤很重要
文件摘要器 原始文件可能 20 頁長——指令不能被淹沒在裡面
程式碼 review 助手 對人來說程式碼和文件長得不一樣，對 token 流來說很像
飲食/訓練計畫生成器 使用者的檔案有很多欄位，需要被當成一組整體考慮
客服工單分類 工單內容和 metadata 要和分類指令分開
多變數模板 任何內插 2+ 變數的 slash command 或模板

什麼時候 XML 是殺雞用牛刀

情境 更好的做法
一行翻譯或改寫 純文字 prompt 即可
自由對話 使用者訊息就讓它是使用者訊息
Eval 結果沒標籤也 OK 的 prompt 不要為了結構而結構

四個產品好處

  1. 可靠性——同一個模板在各種差異很大的使用者輸入下都表現一致，因為「使用者送進來的內容」和「我們告訴 Claude 要做什麼」的邊界永遠不會跑掉。
  2. 更安全的資料處理——使用者內容被隔離在一個有標籤的區塊裡，你就有個天然的地方可以 sanitize、截斷或 log。
  3. 降低 prompt injection 攻擊面——指令在標籤外，內容在標籤內。想注入新指令的攻擊者得先跳出標籤才行。
  4. 可維護性——讀到模板的工程師一眼就能看出哪個變數放哪裡，就像 UI 上有 label 的表單。

PM 決策框架

在 PRD 或設計 review 中審視一個 prompt 模板時，問自己：

問題 如果是「是」 行動
這個模板有內插使用者內容嗎？ 是 用描述性 XML 標籤包起來
有兩種以上類型的輸入嗎（code + docs、data + query）？ 是 每種類型各自有標籤
內插的內容可能很長嗎（整份文件）？ 是 標籤是必備——指令不能被淹沒
指令短、內容長嗎？ 是 標籤讓指令在「頂層」可見
功能現在沒加標籤 eval 就過了嗎？ 是 不要過度工程

PM 常見錯誤

  1. 把 XML 當成純工程議題——PM 應該在 PRD 裡規範標籤結構。標籤名稱是產品契約的一部分，不是實作細節。
  2. 用太泛的標籤名稱——<data> 什麼都沒說。<customer_review> 告訴 Claude「這是評論，請用情感推理」。看到模糊的名字要 push back。
  3. Prompt 改版時忘了加標籤——當模板新增一個內插欄位（例如 user_history），它也需要自己的標籤。這在模板成長過程中很容易 regress。
  4. 沒有做前後 eval 比較——XML 是第一道可靠性修法，但仍需要 eval 資料證明它對你的任務有幫助。
  5. 把輸入標籤誤當成輸出格式——在輸入加標籤不會自動讓 Claude 回 XML。要 XML 輸出得另外下指令。

Key Insight

XML 標籤是 prompt engineering 裡的「有 label 的表單欄位」。對 PM 而言，它把一個不透明的字串轉成一個有命名 slot 的結構化模板——是你可以推理、review 和演進的東西。CCA 考試的觸發詞是「Claude 把指令和內容搞混」或「prompt 內插大量資料」，答案永遠是「加有描述性名稱的 XML 標籤」。

CCA 考試相關性

  • D3（Evaluation & Iteration）：當 prompt 因為邊界模糊失敗時，XML 標籤是第一道改善手段。
  • D2（Tool Design）：同樣的「命名通道」原則也適用於 tool 的輸入和結果。
  • 注意情境描述裡出現「large amount of context」「mixing code and documentation」「interpolating multiple variables」——這些都指向 XML 標籤。

Flashcards

正面 反面
一句話解釋 XML 標籤對 prompt 的作用？ 在 prompt 裡建立有標示的邊界，讓 Claude 知道哪些 token 是資料、哪些是指令。
XML 標籤的檔案櫃比喻是什麼？ 標籤是抽屜上的 label——Claude 打開需要的抽屜，而不是在一大堆東西裡翻。
舉兩個 XML 標籤最有用的產品情境 文件摘要器（大量 context）和程式碼 review 助手（混合內容類型）。
什麼叫「好」的 XML 標籤名稱？ 描述性且具體——<sales_records> 或 <athlete_information>，不是 <data>。
為什麼 XML 標籤可以降低 prompt injection 風險？ 使用者內容被隔離在標籤區塊內，標籤外的指令較難被覆寫。
XML 標籤會改變輸出格式嗎？ 不會——它只管輸入結構；輸出格式需要另外的指令。
PM 什麼時候不該堅持加 XML 標籤？ 一行簡單 prompt，或 eval 沒加標籤就過得很好的功能。
PRD 裡的標籤名稱應該由誰負責？ PM——標籤名稱是產品契約的一部分，不是實作細節。
