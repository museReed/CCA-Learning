一句话总结

XML 标签是「把用户数据安全塞进 prompt」的产品级做法——它让 AI 继续听你的指令，而不是把用户输入误解成新的命令。

PM 为什么要关心 XML 标签

每一个把用户数据插入 prompt 的 AI 功能都共享同一种失败模式：用户内容会和你的指令糊在一起，Claude 做出错误行为。XML 标签是最省力、最高 leverage 的修法。

用户内容类型 没有标签 有标签
长文档 Claude 把文档当成指令的一部分 文档被关在 <document> 内作为数据分析
代码片段 Claude 把代码语法当成自然语言指令 代码关在 <my_code> 内作为数据 debug
多字段的表单 字段值混进散文 所有字段被放进一个 <athlete_information> 容器
混合输入（code + docs） Claude 无法区分 每种类型各自有标签

这跟产品表单要加 label 是同一个道理：告诉用户（或 AI）每一格字段是做什么用的。

心智模型：贴了标签的文件柜抽屉

想象你把 20 页销售记录、5 页公司政策和一个问题堆在同事桌上，一起递过去。你大概会得到一句：「这是什么？」

现在把同样的材料放进一个有三个贴了标签的抽屉的文件柜：销售记录、公司政策、问题。你的同事会打开「问题」抽屉、读完、然后只拉出他需要的那几个抽屉。

XML 标签就是抽屉上的标签。<sales_records>...</sales_records> 在 prompt 里就是一格抽屉。AI 知道要拉哪一格、哪一格可以先放着。

产品使用情境

XML 标签最有用的场景

情境 为什么标签很重要
文档摘要器 原始文档可能 20 页长——指令不能被淹没在里面
代码 review 助手 对人来说代码和文档长得不一样，对 token 流来说很像
饮食/训练计划生成器 用户的档案有很多字段，需要被当成一组整体考虑
客服工单分类 工单内容和 metadata 要和分类指令分开
多变量模板 任何插入 2+ 变量的 slash command 或模板

什么时候 XML 是杀鸡用牛刀

情境 更好的做法
一行翻译或改写 纯文本 prompt 即可
自由对话 用户消息就让它是用户消息
Eval 结果没标签也 OK 的 prompt 不要为了结构而结构

四个产品好处

  1. 可靠性——同一个模板在各种差异很大的用户输入下都表现一致，因为「用户送进来的内容」和「我们告诉 Claude 要做什么」的边界永远不会跑掉。
  2. 更安全的数据处理——用户内容被隔离在一个有标签的区块里，你就有个天然的地方可以 sanitize、截断或 log。
  3. 降低 prompt injection 攻击面——指令在标签外，内容在标签内。想注入新指令的攻击者得先跳出标签才行。
  4. 可维护性——读到模板的工程师一眼就能看出哪个变量放哪里，就像 UI 上有 label 的表单。

PM 决策框架

在 PRD 或设计 review 中审视一个 prompt 模板时，问自己：

问题 如果是「是」 行动
这个模板有插入用户内容吗？ 是 用描述性 XML 标签包起来
有两种以上类型的输入吗（code + docs、data + query）？ 是 每种类型各自有标签
插入的内容可能很长吗（整份文档）？ 是 标签是必备——指令不能被淹没
指令短、内容长吗？ 是 标签让指令在「顶层」可见
功能现在没加标签 eval 就过了吗？ 是 不要过度工程

PM 常见错误

  1. 把 XML 当成纯工程议题——PM 应该在 PRD 里规范标签结构。标签名称是产品契约的一部分，不是实现细节。
  2. 用太泛的标签名称——<data> 什么都没说。<customer_review> 告诉 Claude「这是评论，请用情感推理」。看到模糊的名字要 push back。
  3. Prompt 改版时忘了加标签——当模板新增一个插入字段（例如 user_history），它也需要自己的标签。这在模板成长过程中很容易 regress。
  4. 没有做前后 eval 比较——XML 是第一道可靠性修法，但仍需要 eval 数据证明它对你的任务有帮助。
  5. 把输入标签误当成输出格式——在输入加标签不会自动让 Claude 回 XML。要 XML 输出得另外下指令。

Key Insight

XML 标签是 prompt engineering 里的「有 label 的表单字段」。对 PM 而言，它把一个不透明的字符串转成一个有命名 slot 的结构化模板——是你可以推理、review 和演进的东西。CCA 考试的触发词是「Claude 把指令和内容搞混」或「prompt 插入大量数据」，答案永远是「加有描述性名称的 XML 标签」。

CCA 考试相关性

  • D3（Evaluation & Iteration）：当 prompt 因为边界模糊失败时，XML 标签是第一道改善手段。
  • D2（Tool Design）：同样的「命名通道」原则也适用于 tool 的输入和结果。
  • 注意情境描述里出现「large amount of context」「mixing code and documentation」「interpolating multiple variables」——这些都指向 XML 标签。

Flashcards

正面 反面
一句话解释 XML 标签对 prompt 的作用？ 在 prompt 里建立有标示的边界，让 Claude 知道哪些 token 是数据、哪些是指令。
XML 标签的文件柜比喻是什么？ 标签是抽屉上的 label——Claude 打开需要的抽屉，而不是在一大堆东西里翻。
举两个 XML 标签最有用的产品情境 文档摘要器（大量 context）和代码 review 助手（混合内容类型）。
什么叫「好」的 XML 标签名称？ 描述性且具体——<sales_records> 或 <athlete_information>，不是 <data>。
为什么 XML 标签可以降低 prompt injection 风险？ 用户内容被隔离在标签区块内，标签外的指令较难被覆盖。
XML 标签会改变输出格式吗？ 不会——它只管输入结构；输出格式需要另外的指令。
PM 什么时候不该坚持加 XML 标签？ 一行简单 prompt，或 eval 没加标签就过得很好的功能。
PRD 里的标签名称应该由谁负责？ PM——标签名称是产品契约的一部分，不是实现细节。
