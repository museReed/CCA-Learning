一句话总结

给 Claude 看几个「你要的结果长什么样」的示例，是 PM 角度最接近「训练模型」的动作——它把模糊的质量标准变成可重现的行为，而且直接把你的 eval 数据当成燃料。

PM 为什么要关心示例

每个 PM 都有过这种对话：「输出大概对，但不是我们的语气」或「简单 case 会过，但难 case 会挂」。用文字去描述「我们的语气」或「那些难 case」其实很难。示范 2–4 个示例常常比花一周调指令更快、更稳。

课程原文明确指出 few-shot 是「你最常用的 prompt engineering 技巧之一」。它也是和 eval 反馈循环最紧的技巧——你 eval 里最高分的输出会直接变成下一版 prompt 的示例。

心智模型：带 worked example 的风格指南

想想设计系统或品牌语气 doc 实际怎么运作。文字规则有用（语气：自信但温暖；使用主动语态），但团队真正会去复制的是 worked example：

没有示例 有示例
「要简洁」 「不要这样写：『我们很高兴宣布…』；要这样写：『今日新上线：』」
「要有同理心」 并排放好的退款信和坏版本
「遵守我们的格式」 填好的模板和空白模板并排

Few-shot prompt 对 Claude 做的是同一件事。与其在 PRD 里争论「简洁」的定义，不如把好示例和坏示例直接放进 prompt。AI 每次调用都会读到。

产品使用情境

示例是对的选择

情境 为什么示例有用
含讽刺的情感分析 讽刺太依赖 context，没办法用文字描述——一个示例就搞定
特定 JSON 或 schema 输出 给一个填好的示例比列字段规则更有效
对齐品牌语气 一个 on-brand 的示例胜过一段语气指南
从杂乱文字中抽结构化信息 示例让输出形状不再模糊
已经有 eval 产出「金标」输出的任务 把那些输出升级到 prompt 里当示例

示例是杀鸡用牛刀

情境 更好的做法
很简单的转换 清楚的指令就够
高度变化的创意输出 示例会过度限制模型
没有「标准答案」的任务 示例会暗示一条可能不存在的正确路径

四个产品好处

  1. Edge case 的一致性——功能最容易出糗的地方（讽刺、corner case、模糊输入）正好是示例最能发挥的地方。
  2. 更精准的质量标准——与其在 prompt 里写「要高质量」，不如给 Claude 三个高质量输出。标准变成具体的。
  3. Eval 驱动的改进——你的 eval harness 产出下一版 prompt 的原料。把高分 eval 输出丢回 prompt 是免费的质量升级。
  4. 更少的文字指令——难以用英文描述的复杂规则可以被几个示例取代，而示例比散文更容易 review 和修改。

Eval-to-Example 循环

这是 Lesson 29 对 PM 最重要的收获：

  1. 用目前的 prompt 跑 eval set
  2. 找出得分最高（接近满分）的输出
  3. 把那些 input/output pair 贴进 prompt 当 few-shot 示例
  4. 重跑 eval——难 case 应该会改善，因为 Claude 现在有了「金标」参考行为

这就是为什么 Chapter 03（Prompt Engineering）和 Evaluation chapter 绑在一起。把 prompt 当成静态字符串的 PM 会错过这个循环；把 eval 当成 prompt 更新原料的 PM 会拿到复利式的改进。

PM 决策框架

审视 prompt 模板时问自己：

问题 如果是「是」 行动
任务有难以描述的 corner case 吗？ 是 加覆盖那些 case 的示例
任务需要特定输出形状吗？ 是 至少展示一个完整填好的示例
这个任务已经有高分 eval 输出吗？ 是 把它们升级进 prompt
风格/语气/声音很难用文字定下来吗？ 是 给一个理想示例，而不是规则
任务很简单、没示例也过得去吗？ 是 不要加

PM 常见错误

  1. 把示例当成「工程师的事」——示例编码了产品决策（语气、edge case 处理）。PM 应该拥有「要哪些示例进 prompt」这个决定。
  2. 只挑漂亮的示例——few-shot 的整个价值就是覆盖失败模式。不要只展示 Claude 已经赢的 case。
  3. 让示例变 stale——eval set 会演进，few-shot 示例也要跟着演进。把更新节奏排进功能的维护 rhythm。
  4. 忘了「为什么这是好的」的注解——每个理想输出后面的一行理由会帮助 Claude generalize。PM 常常漏掉，因为看起来像文档而不像 prompt。
  5. 示例没包在 XML 标签里——没有结构，Claude 可能会把示例输入当成真正的任务。PM 应该坚持 <sample_input> / <ideal_output> 的包法。

Key Insight

对 PM 而言，few-shot prompting 是从「我们有个衡量质量的 eval」到「我们有个命中质量标准的 prompt」最短的路径。你 eval 里最高分的输出就是下一版 prompt 的示例。CCA 考试问「如何迭代 prompt 可靠性」时就是在考这个循环——而这是 PM 不用写代码就能驱动的循环。

CCA 考试相关性

  • D3（Evaluation & Iteration）：当 eval 显示特定 case 失败时，few-shot 是经典解法。「把高分输出升级进 prompt」的模式可直接考。
  • D2（Tool Design）：示例和 XML 标签内容块以及严格输出格式紧密搭配。
  • 考试情境：「Claude 在讽刺上失败」「输出格式每次调用都不同」「我们有 eval 分数但 prompt 没改善」——都指向 few-shot 示例。

Flashcards

正面 反面
一句话解释 few-shot prompting 是什么？ 在 prompt 里给 Claude 看输入/输出示例对，以引导响应。
PM 角度，给示例而不是写规则的好处是什么？ 它用具体示范取代难以描述的质量标准（语气、edge case），Claude 可以复制和 generalize。
PM 应该从哪里取得好示例？ 从功能 eval set 中得分最高的输出。
Eval-to-example 循环是什么？ 跑 eval → 找最高分输出 → 贴进 prompt 当示例 → 重跑 eval。
为什么每个理想示例要加「为什么这是好的」注解？ 帮助 Claude 学到判断标准，而不只是表面形式——促进 generalization。
PM 什么时候不该推加更多示例？ 任务简单且已经过 eval 时，或会过度限制创意时。
