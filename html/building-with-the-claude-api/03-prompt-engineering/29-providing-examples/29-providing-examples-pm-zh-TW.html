一句話總結

給 Claude 看幾個「你要的結果長什麼樣」的範例，是 PM 角度最接近「訓練模型」的動作——它把模糊的品質標準變成可重現的行為，而且直接把你的 eval 資料當成燃料。

PM 為什麼要關心範例

每個 PM 都有過這種對話：「輸出大概對，但不是我們的語氣」或「簡單 case 會過，但難 case 會掛」。用文字去描述「我們的語氣」或「那些難 case」其實很難。示範 2–4 個範例常常比花一週調指令更快、更穩。

課程原文明確指出 few-shot 是「你最常用的 prompt engineering 技巧之一」。它也是和 eval 回饋迴圈最緊的技巧——你 eval 裡最高分的輸出會直接變成下一版 prompt 的範例。

心智模型：帶 worked example 的風格指南

想想設計系統或品牌語氣 doc 實際怎麼運作。文字規則有用（語氣：自信但溫暖；使用主動語態），但團隊真正會去複製的是 worked example：

沒有範例 有範例
「要簡潔」 「不要這樣寫：『我們很高興宣布…』；要這樣寫：『今日新上線：』」
「要有同理心」 並排放好的退款信和壞版本
「遵守我們的格式」 填好的模板和空白模板並排

Few-shot prompt 對 Claude 做的是同一件事。與其在 PRD 裡爭論「簡潔」的定義，不如把好範例和壞範例直接放進 prompt。AI 每次呼叫都會讀到。

產品使用情境

範例是對的選擇

情境 為什麼範例有用
含諷刺的情感分析 諷刺太依賴 context，沒辦法用文字描述——一個範例就搞定
特定 JSON 或 schema 輸出 給一個填好的範例比列欄位規則更有效
對齊品牌語氣 一個 on-brand 的範例勝過一段語氣指南
從雜亂文字中抽結構化資訊 範例讓輸出形狀不再模糊
已經有 eval 產出「金標」輸出的任務 把那些輸出升級到 prompt 裡當範例

範例是殺雞用牛刀

情境 更好的做法
很簡單的轉換 清楚的指令就夠
高度變化的創意輸出 範例會過度限制模型
沒有「標準答案」的任務 範例會暗示一條可能不存在的正確路徑

四個產品好處

  1. Edge case 的一致性——功能最容易出糗的地方（諷刺、corner case、模糊輸入）正好是範例最能發揮的地方。
  2. 更精準的品質標準——與其在 prompt 裡寫「要高品質」，不如給 Claude 三個高品質輸出。標準變成具體的。
  3. Eval 驅動的改進——你的 eval harness 產出下一版 prompt 的原料。把高分 eval 輸出丟回 prompt 是免費的品質升級。
  4. 更少的文字指令——難以用英文描述的複雜規則可以被幾個範例取代，而範例比散文更容易 review 和修改。

Eval-to-Example 迴圈

這是 Lesson 29 對 PM 最重要的收穫：

  1. 用目前的 prompt 跑 eval set
  2. 找出得分最高（接近滿分）的輸出
  3. 把那些 input/output pair 貼進 prompt 當 few-shot 範例
  4. 重跑 eval——難 case 應該會改善，因為 Claude 現在有了「金標」參考行為

這就是為什麼 Chapter 03（Prompt Engineering）和 Evaluation chapter 綁在一起。把 prompt 當成靜態字串的 PM 會錯過這個迴圈；把 eval 當成 prompt 更新原料的 PM 會拿到複利式的改進。

PM 決策框架

審視 prompt 模板時問自己：

問題 如果是「是」 行動
任務有難以描述的 corner case 嗎？ 是 加覆蓋那些 case 的範例
任務需要特定輸出形狀嗎？ 是 至少展示一個完整填好的範例
這個任務已經有高分 eval 輸出嗎？ 是 把它們升級進 prompt
風格/語氣/聲音很難用文字定下來嗎？ 是 給一個理想範例，而不是規則
任務很簡單、沒範例也過得去嗎？ 是 不要加

PM 常見錯誤

  1. 把範例當成「工程師的事」——範例編碼了產品決策（語氣、edge case 處理）。PM 應該擁有「要哪些範例進 prompt」這個決定。
  2. 只挑漂亮的範例——few-shot 的整個價值就是覆蓋失敗模式。不要只展示 Claude 已經贏的 case。
  3. 讓範例變 stale——eval set 會演進，few-shot 範例也要跟著演進。把更新節奏排進功能的維護 rhythm。
  4. 忘了「為什麼這是好的」的註解——每個理想輸出後面的一行理由會幫助 Claude generalize。PM 常常漏掉，因為看起來像文件而不像 prompt。
  5. 範例沒包在 XML 標籤裡——沒有結構，Claude 可能會把範例輸入當成真正的任務。PM 應該堅持 <sample_input> / <ideal_output> 的包法。

Key Insight

對 PM 而言，few-shot prompting 是從「我們有個衡量品質的 eval」到「我們有個命中品質標準的 prompt」最短的路徑。你 eval 裡最高分的輸出就是下一版 prompt 的範例。CCA 考試問「如何迭代 prompt 可靠性」時就是在考這個迴圈——而這是 PM 不用寫程式就能驅動的迴圈。

CCA 考試相關性

  • D3（Evaluation & Iteration）：當 eval 顯示特定 case 失敗時，few-shot 是經典解法。「把高分輸出升級進 prompt」的模式可直接考。
  • D2（Tool Design）：範例和 XML 標籤內容區塊以及嚴格輸出格式緊密搭配。
  • 考試情境：「Claude 在諷刺上失敗」「輸出格式每次呼叫都不同」「我們有 eval 分數但 prompt 沒改善」——都指向 few-shot 範例。
