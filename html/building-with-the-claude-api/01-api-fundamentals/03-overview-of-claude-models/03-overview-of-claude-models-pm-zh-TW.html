One-Liner

Claude 提供三個模型層級——Opus（最聰明）、Sonnet（平衡）、Haiku（最快/最便宜）——為每個功能選擇正確的模型是 PM 對 AI 成本和使用者體驗影響最大的槓桿。

心智模型：顧問公司

把三個模型想成三個等級的顧問：

模型 顧問類比 什麼時候聘用
Opus 資深合夥人——深度專業、價格昂貴、需要時間 複雜策略工作、高風險決策
Sonnet 中階經理——能力強、價格合理、速度穩定 大多數日常客戶工作
Haiku 初級分析師——快速、便宜、擅長例行任務 大量例行工作、快速周轉

你不會派資深合夥人去回覆例行郵件。同理，你不該把簡單分類任務路由到 Opus。

三個模型一覽

維度 Opus Sonnet Haiku
智能 最高 高 中等
速度 較慢 中等 最快
成本 最高 中等 最低
延伸思考 支援 支援 不支援
最適合 複雜推理、多小時自主任務 通用、編碼、大多數功能 即時聊天、分類、大量處理

為什麼模型選擇是 PM 的決策

模型選擇不純粹是技術問題。它直接控制：

商業槓桿 影響
每次互動成本 同一段對話，Opus 可能比 Haiku 貴 5-10 倍
使用者感知延遲 Haiku 不到一秒回應；Opus 可能需要幾秒鐘
功能品質天花板 Haiku 不能做延伸思考；某些功能沒有 Opus/Sonnet 根本不可能
可擴展性預算 大量功能跑 Opus 幾小時就能燒完預算

PM 的工作：定義哪些功能需要哪個品質層級，然後與工程團隊合作做相應路由。

產品使用場景

場景 建議模型 原因
客服聊天機器人（大量） Haiku 速度 + 成本；大多數客服查詢都是例行的
法律審查的文件分析 Opus 對長文件的複雜推理
應用內寫作助手 Sonnet 品質好、延遲可接受
即時自動完成建議 Haiku 亞秒級延遲不可妥協
多步驟研究代理 Opus 需要延伸思考和自主性
程式碼審查工具 Sonnet 強大的編碼能力、精準編輯

PM 決策框架

問題 答案決定什麼
我們的功能需要多複雜的推理？ Opus vs Sonnet vs Haiku
回應對使用者來說必須多快？ 即時用 Haiku，非同步用 Sonnet/Opus
每個使用者每月的成本上限是多少？ 模型混合比例
我們能對不同功能使用不同模型嗎？ 幾乎總是可以——這是建議的模式
有任何功能需要延伸思考嗎？ 如果是，Haiku 在該功能上就被排除

多模型架構

最具成本效益的生產模式是使用多個模型：

應用層級 模型 PM 理由
問候 / 路由 / 分類 Haiku 即時、便宜；使用者在這裡不需要智能
主要功能邏輯 Sonnet 品質重要，回應時間也重要
升級 / 複雜邊緣案例 Opus 把昂貴的模型留給高價值時刻

與所有事情都用單一高階模型相比，這種模式可以減少 60-80% 的 AI 成本。

常見 PM 錯誤

  1. 預設使用「最好的」模型 —— 所有事情都用 Opus 就像請資深合夥人來歸檔。大多數功能不需要最高智能。
  2. 忽視每次互動的成本計算 —— 承諾之前先算數字。一個每天服務 10 萬使用者的功能，用 Opus 和用 Haiku 的成本天差地別。
  3. PRD 中沒有為每個功能指定模型 —— 如果把模型選擇丟給工程團隊而不給脈絡，他們要麼超支（為安全起見用 Opus）、要麼交付不足（為預算用 Haiku）。
  4. 忘記 Haiku 的限制 —— 它不能做延伸思考。如果你的功能規格寫「AI 應該逐步推理」，Haiku 就出局了。
  5. 把模型選擇當成永久決定 —— 你可以隨著品質或預算需求演變而更換每個功能的模型。從第一天就建好路由彈性。

核心洞察

模型選擇是 PM 在 Claude 驅動產品上做的最高槓桿成本決策。它不是可以委派的技術細節——而是同時決定你的單位經濟學和使用者體驗的商業架構選擇。正確答案幾乎總是「使用多個模型，按任務複雜度路由」。

CCA 考試相關性

  • D5（企業部署）：模型選擇題幾乎必考。預期會描述一個工作負載的情境，要求你選擇合適的模型。
  • 記住三方取捨：智能 vs 速度 vs 成本。
  • 記住：Haiku 不支援延伸思考。這是最容易出題的差異點。
