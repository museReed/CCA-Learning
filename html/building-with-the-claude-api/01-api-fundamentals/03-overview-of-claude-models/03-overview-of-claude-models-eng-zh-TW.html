One-Liner

Claude 提供三個模型家族——Opus（最高智能）、Sonnet（平衡）、Haiku（最快/最便宜）——為每個功能選擇正確的模型是任何 Claude 驅動系統的首要架構決策。

三大模型家族

三者共享相同的核心能力（文字生成、程式碼撰寫、圖片分析）。差異在於最佳化目標：

智能 ◀─────────────────────────────────────▶ 速度 / 成本

   Opus              Sonnet               Haiku
   ├── 最高智能      ├── 平衡              ├── 最快
   ├── 推理能力      ├── 強大編碼能力      ├── 最低成本
   ├── 多小時        ├── 精準的程式碼      ├── 不支援延伸思考
   │   自主運作      │   庫編輯            │
   ├── 較高成本      ├── 中等成本          ├── 即時 UX
   └── 較高延遲      └── 中等延遲          └── 最低延遲

Opus —— 最大智能

屬性 細節
最佳化方向 最高智能 / 推理能力
最適合 複雜的多步驟任務、長時間自主工作階段（數小時）、深度規劃
推理 支援延伸思考——可以在困難問題上花時間思考
取捨 較高延遲、較高每次請求成本
使用時機 品質比速度重要；複雜需求需要深度推理

Sonnet —— 甜蜜點

屬性 細節
最佳化方向 智能、速度與成本的平衡
最適合 大多數實用場景、編碼任務、精準的程式碼庫編輯
推理 支援延伸思考
取捨 既不是最聰明也不是最快，但是最好的預設選擇
使用時機 通用工作；本課程的預設模型

Haiku —— 速度冠軍

屬性 細節
最佳化方向 最快回應時間、最低成本
最適合 面向使用者的即時互動、大量批次處理
推理 不支援延伸思考
取捨 與 Opus/Sonnet 相比智能較低
使用時機 延遲敏感的功能、分類、簡單擷取

模型選擇框架

決策樹很簡單：

def select_model(task):
    if task.requires_deep_reasoning or task.is_multi_step_complex:
        return "opus"
    elif task.requires_real_time or task.is_high_volume:
        return "haiku"
    else:
        return "sonnet"  # 大多數應用的預設

關鍵洞察：生產系統通常同時使用多個模型：

層級 模型 理由
面向使用者的聊天 Haiku 速度對 UX 至關重要
核心業務邏輯 Sonnet 品質與成本的平衡
複雜推理任務 Opus 品質優先於一切

這不是過早最佳化——而是成本架構。全部走 Opus 的系統成本會比用 Haiku 處理簡單任務的系統高 5-10 倍。

推理能力比較

能力 Opus Sonnet Haiku
標準回應 是 是 是
延伸思考 是 是 否
自適應（簡單→快速，複雜→思考） 是 是 否

Haiku 不支援延伸思考是最重要的技術差異點。如果你的功能需要鏈式推理，Haiku 不是選項。

常見錯誤

  1. 所有事情都用 Opus —— 浪費金錢又增加延遲；大多數任務不需要頂尖智能。
  2. 用 Haiku 做複雜推理 —— Haiku 不支援延伸思考；在困難問題上只會給出膚淺的答案。
  3. 選一個模型就不再改變 —— 模型選擇應該是每個功能獨立決定的，不是整個應用一致的。
  4. 忽視成本倍數差異 —— Haiku 和 Opus 的定價差距意味著模型選擇是預算決策，不只是技術決策。
  5. 假設模型名稱是固定的 —— 永遠使用具體的模型字串（如 claude-sonnet-4-5），不要假設「Sonnet」永遠代表同一個版本。

核心洞察

模型選擇不是一次性的決策——它是一種路由策略。最好的生產系統把模型選擇當作參數而非常數。它們把簡單任務路由到 Haiku、標準任務路由到 Sonnet、複雜任務路由到 Opus。這是你做的第一個架構決策，直接控制你的成本和延遲預算。

CCA 考試相關性

  • D5（企業部署）：預期會出情境題，例如「大量客服機器人該用哪個模型？」（Haiku）或「多步驟研究代理該用哪個？」（Opus）。
  • 模型選擇題目可能會呈現取捨情境，要求你選擇合適的模型家族。
  • 記住 Haiku 不支援延伸思考——這是考試很可能考的差異點。
