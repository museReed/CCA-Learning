One-Liner

Claude 提供三个模型家族——Opus（最高智能）、Sonnet（平衡）、Haiku（最快/最便宜）——为每个功能选择正确的模型是任何 Claude 驱动系统的首要架构决策。

三大模型家族

三者共享相同的核心能力（文本生成、代码编写、图片分析）。差异在于优化目标：

智能 ◀─────────────────────────────────────▶ 速度 / 成本

   Opus              Sonnet               Haiku
   ├── 最高智能      ├── 平衡              ├── 最快
   ├── 推理能力      ├── 强大编码能力      ├── 最低成本
   ├── 多小时        ├── 精准的代码        ├── 不支持扩展思考
   │   自主运行      │   库编辑            │
   ├── 较高成本      ├── 中等成本          ├── 实时 UX
   └── 较高延迟      └── 中等延迟          └── 最低延迟

Opus —— 最大智能

属性 细节
优化方向 最高智能 / 推理能力
最适合 复杂的多步骤任务、长时间自主工作会话（数小时）、深度规划
推理 支持扩展思考——可以在困难问题上花时间思考
取舍 较高延迟、较高每次请求成本
使用时机 质量比速度重要；复杂需求需要深度推理

Sonnet —— 甜蜜点

属性 细节
优化方向 智能、速度与成本的平衡
最适合 大多数实用场景、编码任务、精准的代码库编辑
推理 支持扩展思考
取舍 既不是最聪明也不是最快，但是最好的默认选择
使用时机 通用工作；本课程的默认模型

Haiku —— 速度冠军

属性 细节
优化方向 最快响应时间、最低成本
最适合 面向用户的实时交互、大批量处理
推理 不支持扩展思考
取舍 与 Opus/Sonnet 相比智能较低
使用时机 延迟敏感的功能、分类、简单提取

模型选择框架

决策树很简单：

def select_model(task):
    if task.requires_deep_reasoning or task.is_multi_step_complex:
        return "opus"
    elif task.requires_real_time or task.is_high_volume:
        return "haiku"
    else:
        return "sonnet"  # 大多数应用的默认值

关键洞察：生产系统通常同时使用多个模型：

层级 模型 理由
面向用户的聊天 Haiku 速度对 UX 至关重要
核心业务逻辑 Sonnet 质量与成本的平衡
复杂推理任务 Opus 质量优先于一切

这不是过早优化——而是成本架构。全部走 Opus 的系统成本会比用 Haiku 处理简单任务的系统高 5-10 倍。

推理能力比较

能力 Opus Sonnet Haiku
标准响应 是 是 是
扩展思考 是 是 否
自适应（简单→快速，复杂→思考） 是 是 否

Haiku 不支持扩展思考是最重要的技术差异点。如果你的功能需要链式推理，Haiku 不是选项。

常见错误

  1. 所有事情都用 Opus —— 浪费钱又增加延迟；大多数任务不需要顶尖智能。
  2. 用 Haiku 做复杂推理 —— Haiku 不支持扩展思考；在困难问题上只会给出肤浅的回答。
  3. 选一个模型就不再改变 —— 模型选择应该是每个功能独立决定的，不是整个应用统一的。
  4. 忽视成本倍数差异 —— Haiku 和 Opus 的定价差距意味着模型选择是预算决策，不只是技术决策。
  5. 假设模型名称是固定的 —— 永远使用具体的模型字符串（如 claude-sonnet-4-5），不要假设"Sonnet"永远代表同一个版本。

核心洞察

模型选择不是一次性的决策——它是一种路由策略。最好的生产系统把模型选择当作参数而非常量。它们把简单任务路由到 Haiku、标准任务路由到 Sonnet、复杂任务路由到 Opus。这是你做的第一个架构决策，直接控制你的成本和延迟预算。

CCA 考试相关性

  • D5（企业部署）：预期会出情境题，例如"大量客服机器人该用哪个模型？"（Haiku）或"多步骤研究代理该用哪个？"（Opus）。
  • 模型选择题目可能会呈现取舍情境，要求你选择合适的模型家族。
  • 记住 Haiku 不支持扩展思考——这是考试很可能考的差异点。
