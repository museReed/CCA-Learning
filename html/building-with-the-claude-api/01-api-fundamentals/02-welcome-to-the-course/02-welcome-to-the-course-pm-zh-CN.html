One-Liner

这堂导览课程列出完整学习路线图——从 API 基础到代理工作流——让 PM 清楚看到规划、排序和交付 AI 驱动功能所需的每一项 Claude 能力。

产品能力地图

把每个课程模块想成你的产品团队可以解锁的一项能力：

模块 为产品带来的能力 PM 决策
API 基础 基本 AI 文本生成 "我们能不能调用 Claude？"——基本门槛
提示评估 AI 输出的质量保证 "我们怎么知道 AI 在正常运作？"——最重要的模块
提示工程 控制 AI 语气、格式、准确度 "我们怎么让 AI 做我们想要的事？"
工具使用 Claude 调用外部函数 "Claude 能查数据、订机票吗？"
RAG 让 AI 基于公司文档回答问题 "Claude 能回答关于我们产品的问题吗？"
MCP 与外部服务的标准化集成 "我们怎么把 Claude 接到 Slack、数据库等？"
Claude Code / Computer Use AI 自主操作软件 "AI 能替我们写代码或浏览网页吗？"
工作流与代理 多步骤 AI 流程 "我们能自动化整个工作流，而不只是单一查询吗？"

PM 为什么该在意这份路线图

课程遵循明确的依赖链。这对产品规划至关重要，因为：

  1. 没有工具使用就无法交付代理功能 —— 如果你的路线图把"AI 代理"放在 Q1、"工具集成"放在 Q2，这个计划就是坏的。
  2. 提示评估是质量的把关者 —— 讲师称之为最重要的实践。跳过评估规划的 PM 会得到 demo 很好看、但生产环境失败的 AI 功能。
  3. RAG 解锁企业价值 —— 大多数企业客户都要求 AI 了解他们的数据。RAG 就是实现这件事的地方。

PM 决策框架

要问团队的问题 为什么重要
我们的 API 访问和密钥设好了吗？ 卡在这里 = 工程速度归零
我们在上线前有规划提示评估吗？ 没有评估 = 没有质量保证；讲师最强调这一点
哪个模块对应我们的下一个功能？ 优先学习路线图最需要的模块
我们在构建工具还是代理？ 工具是确定性的、可测试的；代理更灵活但更难评估
我们的使用场景需要 RAG 吗？ 如果用户会问关于专有数据的问题，答案就是需要

常见 PM 错误

  1. 在不了解先决条件的情况下规划 AI 代理 —— 代理需要提示工程 + 工具使用 + 评估。跳过任何一项都会导致不可靠的功能。
  2. 把提示工程当成"工程师的事" —— 提示设计决策就是产品决策（语气、格式、准确度目标）。
  3. 忽略提示评估 —— 讲师明确说过：不做评估就部署，你的用户不会得到你预期的结果。
  4. 一次规划太多能力 —— 依赖链意味着你应该先交付 API 基础、再交付工具、再交付代理。
  5. 没有为迭代留出时间 —— 课程强调扩展、修改和破坏 Notebook。你的冲刺计划需要留出提示迭代的空间。

核心洞察

这份路线图不只是课程大纲——它是 AI 功能的产品成熟度模型。公司会按这些阶段推进：基本 API 调用、然后评估、提示工程、工具使用、RAG、MCP，最后才是代理。试图跳过阶段，就是 AI 功能在生产环境失败的原因。

CCA 考试相关性

  • D1（代理式编码基础）：课程弧线直接对应 CCA 技能进程——考试题目假设你熟悉每个模块。
  • D5（企业部署模式）：先决条件（API 密钥设置、环境准备）是企业部署的基础。
  • 考试预期你理解依赖链：不了解工具使用基础就无法回答代理相关问题。

Flashcards
