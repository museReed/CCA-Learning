The Full RAG Flow — 工程深度解析

项目 内容
考试领域 D1 — Agentic Architecture (22%) 主要；D4 — Safety & Alignment (20%) 次要
Task Statements 1.3（context management）、4.1（grounded responses）
来源 building-with-the-claude-api / 05-rag / Lesson 48

一句话总结

完整 RAG flow 是一个六步 pipeline — chunk → embed → 存 → embed query → similarity search → 喂 prompt 给 Claude — 其中 cosine similarity 是把「找相关文字」转换成「找最接近向量」的度量。

六个步骤

  [预处理 — 每份文档跑一次]
  1. 切 source text
  2. 为每个 chunk 产生 embedding
  3. 存进 vector database
                      │
                      ▼
  [查询时 — 每次用户请求跑]
  4. 把用户 query 转 embedding
  5. 找相似 embedding（cosine similarity）
  6. 用 retrieved chunks 组最后 prompt → Claude

步骤 1-3 是预处理，事先跑，每份文档一次。步骤 4-6 是查询时，每次用户请求都跑。这个切割就是 RAG 可扩展的原因：昂贵的工作做一次，便宜的工作每次 query 才做。

Step 1：切 Source Text

把每份文档切成可管理的 chunks。lesson 用两个 toy section：

  • Section 1 — Medical Research："This year saw significant strides in our understanding of XDR-47, a 'bug' we have not seen before."
  • Section 2 — Software Engineering："This division dedicated significant effort to studying various infection vectors in our distributed systems."

注意词汇陷阱：医疗 section 用了「bug」，软件 section 用了「infection vectors」。keyword search 会很开心地把两者接错线。RAG 必须做得比这好。

Step 2：产生 Embeddings

每个 chunk 喂进 embedding model（本课程是 VoyageAI），拿回一个数值向量。

lesson 用玩具二维模型让几何看得见。在这个想象模型里：

  • 维度 1 = 「文字多谈医疗领域」
  • 维度 2 = 「文字多谈软件工程」

医疗 chunk 的 embedding 是 [0.97, 0.34] — 很医疗，但「bug」这个字造成一点软件词汇渗漏。软件 chunk 是 [0.30, 0.97] — 偏重软件，但「infection vectors」带进一点医疗信号。

真正的模型用数百或数千维度，无法按维度解读。玩具模型存在只是为了说明 similarity 在几何上长什么样子。

Normalization（归一化）

大部分 embedding API 会把向量归一化成单位长度（magnitude 1.0）。数学自动处理，你通常不用自己做。归一化后：

  • [0.97, 0.34] → [0.944, 0.331]
  • [0.30, 0.97] → [0.295, 0.955]

你可以把归一化向量想成单位圆（或高维的单位超球面）上的点。归一化让 cosine similarity 和 dot product 等价，也是 vector database 能优化 similarity search 的原因。

Step 3：存进 Vector Database

Vector database 是优化过的专门存储库：

  • 存储长长的 float list
  • 搜索最接近 query 向量的向量（approximate nearest neighbor）
  • 扩展到百万或十亿级向量

这步完成后预处理就结束。pipeline 暂停，等用户 query。到这里为止的所有工作都会被未来所有 query 摊提。

Step 4：处理 User Query

当用户问「I'm curious about the company. In particular, what did the software engineering dept do this year?」，你把 query 喂进同一个 embedding model（与 chunks 用同一个）。

玩具示例给的原始向量是 [0.1, 0.89] — 低医疗、高软件工程。归一化后：[0.112, 0.993]。

重点：同一个模型。而且若 provider 支持，这里用 input_type="query"（Step 2 的 chunks 则用 input_type="document"）。

Step 5：用 Cosine Similarity 找相似 Embedding

把 query 向量送到 vector database，要求返回最接近的存储向量。

Cosine Similarity 怎么运作

Cosine similarity 衡量两个向量之间夹角的余弦：

  • 范围：-1 到 +1
  • 接近 +1 → 高相似度（向量几乎同方向）
  • 接近 -1 → 非常不同（向量反方向）
  • 0 → 垂直（没有有意义的关联）

玩具示例里：

比对 Cosine Similarity
Query vs. 软件 chunk 0.983（非常高）
Query vs. 医疗 chunk 0.398（低很多）

Vector database 返回软件 chunk — 正是用户想要的，尽管两个 chunks 有重叠词汇（「bug」、「infection vectors」）。

Cosine Distance

你也会在 vector database 文档看到 cosine distance，定义是 1 - cosine_similarity。用 cosine distance：

  • 接近 0 → 高相似度
  • 较大值 → 较不相似

这通常比较好解读，因为「距离」会随东西越不相似而变大。同样的底层数学，只是报告惯例不同。

Step 6：组最后 Prompt

把用户问题和 retrieved chunk(s) 组成 prompt 送给 Claude：

Answer the user's question about the financial document.

<user_question>
How many bugs did engineers fix this year?
</user_question>

<report>
## Section 2: Software Engineering
This division dedicated significant effort to studying various infection vectors in our distributed systems
</report>

Retrieved chunk 用 XML tag（<report>）包起来，让 Claude 知道哪一部分是用户问题、哪一部分是 grounding source。Claude 接着从组合后的 context 产出答案。

这是 retrieval-augmented generation 的「generation」那一半。注意 Claude 自己从未调用 vector database — 你的应用程式做的。从 Claude 角度看，它收到的是一个普通 prompt，只是有异常多的精心挑选 context。

为什么词汇陷阱是重点

「bug」例子特意构造来让 keyword search 出糗。两个 chunks 都含疾病类语言；基于关键词的 retriever 很可能对「how many bugs did engineers fix」返回医疗 chunk。在 semantic embedding 上做 cosine similarity 能正确处理，因为 query 向量方向（软件偏重）接近软件 chunk 的方向。

这就是为什么 embedding 重要：它测的是语义方向，不是词汇重叠。

CCA Task 对应

  • Task 1.3（Context Management） — 整个 RAG flow 就是 context-management pipeline。预处理决定什么可被检索；retrieval 决定什么进 context；prompt 组装决定 Claude 怎么看。
  • Task 4.1（Grounded Responses） — 最终 prompt 把 Claude 的答案 grounded 在 retrieved chunks。用 XML tag 包 chunks 等于向模型表明「这是来源素材」。

常见错误

  1. 忘了 query 要用和 chunks 同一个 embedding model — 向量落在不兼容空间，similarity score 变毫无意义。
  2. 跳过 normalization — provider 没帮你归一化、且 database 用 dot product 时，非归一化向量会产生错误排序。
  3. 返回太多 chunks — top-k 设太大会把低相关内容塞进 prompt，分散 Claude 注意力。
  4. 最终 prompt 没 XML／结构化标记 — Claude 分不出问题和来源素材，grounding 变弱。
  5. 生产没记录 retrieval score — 答案错时需要 retrieved chunks 的 similarity score 来 debug 是 retrieval 问题还是 generation 问题。

关键洞察

六步 RAG flow 其实是两半黏在一起：离线预处理 pipeline（chunk → embed → 存），每份文档跑一次；在线 query pipeline（embed query → retrieve → prompt），每次请求跑。你未来会 debug 的每个 RAG bug 都住在这六步其中之一，所以要知道边界在哪、哪一步拥有哪种失败模式。

CCA 考试关联

  • D1（Agentic Architecture）：端对端 RAG pipeline 是核心 agentic-pattern 题目。预期会看「RAG 步骤正确顺序是什么」或「cosine similarity 在 flow 里的哪一步」这种题。
  • D4（Safety & Alignment）：RAG grounding 降低幻觉。「bug」例子是好 retrieval 阻止自信错答案的教科书案例。
  • 要准备好定义 cosine similarity、它的范围、和 cosine distance 的关系。

Flashcards

Front Back
列出完整 RAG flow 的六个步骤。 1) 切文字、2) 产生 embedding、3) 存进 vector DB、4) Embed user query、5) 找相似 embedding（cosine similarity）、6) 组最终 prompt 调用 Claude。
哪几步是预处理 vs. 查询时？ 1-3 是预处理（每份文档一次）；4-6 是查询时（每次用户请求一次）。
Cosine similarity 的范围？ -1 到 +1。
Cosine similarity 为 +1 代表什么？ 向量同方向 — 最高相似度。
Cosine similarity 为 0 代表什么？ 向量垂直 — 没有有意义的关联。
Cosine distance 怎么定义？ 1 - cosine_similarity。小值 = 高相似度；大值 = 较不相似。
为什么 query 和 chunks 必须用同一个 embedding model？ 否则向量在不同几何空间，similarity 失去意义。
什么是 normalization、为什么重要？ 把向量缩到单位长度（magnitude 1.0）。让 cosine similarity 和 dot product 等价，并支持有效率的 similarity search。
「bug」例子里为什么 cosine similarity 成功而 keyword search 失败？ 它比较语义方向不是词汇重叠，所以 query 的软件偏重方向匹配软件 chunk，即使词汇有重叠。
