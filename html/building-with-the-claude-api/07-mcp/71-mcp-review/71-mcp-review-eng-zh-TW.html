MCP 複習 — 工程師深度解析

項目 細節
考試領域 D2 — 工具設計與 MCP 整合 (18%)
任務陳述 2.1 (MCP 架構)、2.2 (基本元素)、2.3 (tool vs resource vs prompt)
來源 building-with-the-claude-api / 07-mcp / 第 71 課

一句話摘要

MCP 的三個伺服器基本元素 — tools、resources 和 prompts — 各由應用程式的不同部分控制：tools 由模型控制、resources 由應用程式控制、prompts 由使用者控制，這個對應關係告訴你在新增功能時該選擇哪個元素。

三大基本元素與其控制者

基本元素 控制者 用途 範例
Tools 模型 (Claude) 擴展 Claude 的能力 Claude 決定執行 JavaScript 來計算 √3
Resources 應用程式碼 為應用程式/UI 提供資料 從 Google Drive 取得文件列表用於自動完成或上下文增強
Prompts 使用者 預定義工作流程 使用者點擊按鈕或輸入斜線指令啟動工作流程

Tools — 模型控制

Claude 獨自決定何時呼叫工具。你的程式碼定義工具及其 schema，但模型在執行時做出決定。如果你需要為 Claude 新增能力（執行程式碼、查詢資料庫、呼叫 API），就在 MCP 伺服器上實作工具，或透過 MCP 客戶端使用現有伺服器的工具。

Resources — 應用程式控制

應用程式碼決定何時取得資源。資源提供資料 — 它們不執行操作。課程專案中的兩個常見模式：

  1. UI 豐富化 — 取得資源來填充自動完成建議
  2. 提示詞增強 — 取得資源來在發送給 Claude 之前注入上下文

如果你需要將資料放入應用程式以供顯示或提供上下文，使用 resource。

Prompts — 使用者控制

使用者透過點擊 UI 元素或輸入斜線指令來啟動 prompt。Prompts 是預定義、已最佳化的工作流程 — 想想 Claude.ai 聊天輸入框下方的按鈕列。每個按鈕都觸發一個預先撰寫的提示模板。

如果你需要實作使用者啟動的預定義工作流程，使用 prompt。

決策框架

需要為 Claude 新增能力？          → Tool
需要將資料放入應用程式/UI？       → Resource
需要使用者觸發的預定義流程？      → Prompt

這些是高層指導原則，不是硬性規則。關鍵洞察是每個基本元素服務不同的受眾：模型、應用程式或使用者。

CCA 考試關聯

  • D2 (MCP 整合)：預期會出現「你應該在什麼情況下使用哪個 MCP 基本元素？」類型的題目 — 答案直接對應到誰控制該操作（模型/應用程式/使用者）。
  • 注意：「Tools 由模型控制」是最常考的事實。Resources 和 prompts 常被混淆 — 記住 resources 是應用程式碼用來取得資料的，prompts 是使用者啟動的工作流程。

記憶卡

問題 答案
MCP 伺服器的三大基本元素是什麼？ Tools、Resources 和 Prompts。
誰在執行時控制 MCP tools？ 模型 (Claude) — 它決定何時呼叫工具。
誰在執行時控制 MCP resources？ 應用程式碼 — 你的應用程式決定何時取得資料。
誰在執行時控制 MCP prompts？ 使用者 — 他們透過 UI 元素或斜線指令觸發 prompts。
何時該用 tool，何時該用 resource？ Tool：為 Claude 新增能力。Resource：將資料放入應用程式/UI。
Claude.ai 中 resource 的範例是什麼？ 「從 Google Drive 新增」按鈕 — 應用程式取得並顯示文件列表。
Claude.ai 中 prompt 的範例是什麼？ 聊天輸入框下方的預建對話起始按鈕。
這些基本元素-控制者的對應是硬性規則嗎？ 不是 — 它們是高層指導原則，指示每個基本元素的主要目標受眾。
