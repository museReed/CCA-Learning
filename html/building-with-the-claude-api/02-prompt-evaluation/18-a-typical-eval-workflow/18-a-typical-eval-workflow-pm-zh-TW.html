一句話摘要

Prompt eval 工作流是 AI 團隊版的「產品上線 checklist」— 可重複的五步驟流程，產出一個 PM 能寫進發佈文件、OKR、優先排序決策的單一品質數字。

心智模型：食譜試驗廚房

專業試驗廚房不會靠一次試吃就決定要不要推出新款早餐穀片。他們有固定的試吃小組、固定的評分表，然後對每一個版本重複打分直到平均分過門檻。這就是 prompt evaluation 對 prompt 做的事。

試驗廚房 Prompt eval
初版食譜 起草的 prompt
試吃小組 Eval 資料集
試吃員嘗每一匙 餵進 Claude
試吃員依評分表打 1-10 分 餵進 grader
主廚微調食譜、重新評分 迭代 prompt 並重複

資料集就是小組。Grader 就是評分表。Prompt 就是食譜。小組和評分表固定，只動食譜 — 這就是如何隔離「有效改動」的方法。

五個步驟 — PM 速查表

步驟 產出 PM 價值
1. 起草 prompt Baseline 版本 能被量測、有可改進的基準
2. 建立資料集 N 筆真實輸入 定義「品質」在此功能的意義
3. 餵進 Claude N 筆原始輸出 當前行為的證據
4. 餵進 grader 數值分數（每筆 1-10、取平均） 能寫進發佈文件的品質指標
5. 迭代 新 prompt + 新分數 證明團隊真的在改善產品

Grader 吐出來的數字，就是你跟財務、行銷、高層吵架時的武器。沒有第 4 步就沒有數字，沒有數字就沒得吵，沒得吵就沒辦法排優先級。

課程的實例（為什麼這很關鍵）

課程用三個問題的玩具資料集，示範改 prompt 一行，平均分從 7.66 升到 8.7。整個要點濃縮成這件事：

  • 改動前：「這比較好嗎？」→ 意見。
  • 改動後：「平均分從 7.66 升到 8.7。」→ 事實。

你現在可以用一個真實的 delta 更新發佈文件、OKR 儀表板、工程 roadmap。那個數字讓 prompt 工作對組織裡其他人「可見、可讀」。

PM 決策框架

當工程團隊說新版 prompt 好了，請走過這四個問題：

問題 好答案長什麼樣
目前 prompt 的 baseline 分數？ 來自固定資料集的 1-10 分數字
提案 prompt 的新分數？ 來自同一資料集、同一 grader 的數字
資料集跟上次跑比有變過嗎？ 沒 — 若有，比較無效
提升幅度值得上線嗎？ 主觀，但至少大家在討論的是數字

任何一題的答案是「我們沒做」，你就該擋下來。團隊現在要你上線的是意見，不是量測過的改善。

PM 常見錯誤

  1. 接受「我們測過比較好」 — 永遠要問 baseline 數字和新數字。
  2. 讓團隊在兩次跑之間悄悄改資料集 — 這會摧毀可比性，讓所有宣稱的改善失效。
  3. 把 grader 當成「技術細節、PM 不用管」 — Grader 就是你的品質定義；不擁有它等於放棄產品判斷權。
  4. 一升一點點就慶祝「上線」 — 第一個改善很少是最佳的；要求持續迭代直到分數進入平台期。
  5. 忘記把分數寫進發佈文件 — 沒寫下來，組織會忘記「好」的定義是什麼。

Key Insight

五步驟工作流把 prompt engineering 從「相信我們就好」變成「看這個數字」。CCA 考試裡只要 D3 問到流程、可重現性、迭代，答案就是把五個步驟照順序講一遍、並強調資料集保持不變。對 PM 來說，這就是讓你不需要變成 prompt engineer，也能讓工程團隊對 AI 品質負責的機制。

CCA 考試相關性

  • D3（Evaluation & Iteration）：五步驟迴圈是 THE canonical workflow。考題可能把步驟打亂要你排序。
  • D5（Enterprise Deployment）：這條迴圈是「工程覺得好了」與「上線」之間的閘門。
  • 注意「實驗中途換 dataset」的情境 — 正確答案永遠是資料集必須保持不變。
