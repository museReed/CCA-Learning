一句话摘要

Eval 数据集编码了你的 AI 功能"什么叫好" — 最快拿到可用第一版的方法是让 Claude 帮你打草稿，然后把那个草稿当成一份 PM 拥有并持续迭代的活产物。

心智模型：用户问卷

把 eval 数据集想成客户访谈问卷。PM 不会从白纸开始做用户研究 — 他们会先起草问卷、测试几个人、然后精修。数据集的做法完全一样：

问卷概念 Eval 数据集概念
起草问卷题目 用 Haiku 生成初版数据集
找几个人试答 对生成的数据集跑一次 pilot eval
根据信号精修题目 编辑数据集补强薄弱点
锁定最终版 存成 dataset.json 并 commit 进版控
每季度重新跑问卷 每次 prompt 改动都重跑 eval

数据集就是问卷。每一条 entry 都是一个"你产品要好好处理这种输入"的决定。

AWS Code Assistant 示例

课程的 running example 是一个帮用户写 AWS 专属 Python、JSON、regex 的 prompt。要求是干净输出 — 没有前言、没有"这是你的代码"开场白。对 PM 来说，这个设定有三点值得注意：

  1. 输出格式是产品需求，不是技术细节。 干净代码让功能能在复制粘贴工作流里可用；杂乱代码会毁掉整个 use case。
  2. 数据集存在的意义，就是证明 prompt 能在非团队手挑的输入上遵守这个产品需求。
  3. 生成步骤产出的数据集能在客户看到之前就以极低成本暴露格式失败。

构建数据集的两种做法

做法 速度 保真度 何时用
手工构造 慢 高 — 每个案例你都掌控 专业领域、受监管行业、高风险功能
Claude 生成 快 中 — 覆盖取决于 prompt Bootstrapping、通用任务、早期功能

课程选 Claude 生成是有道理的：初始成本极低，而且你永远可以事后手工编辑 — 编辑已生成的数据集比从零写快得多。

成本杠杆： 课程让工程师用 Haiku（不是 Sonnet）来生成。对 PM 来说，这直接对应到运营成本 — 批量数据集工作绝对不该用最贵的模型。

PM 决策框架

当团队说他们生成了一份 eval 数据集，走过这些问题：

问题 好答案长什么样
谁生成的 — 人类还是 Claude？ 都可以，但答案会影响后续问题
如果是 Claude：用 Haiku 还是 Sonnet？ Haiku — 其他都是批量工作上的浪费
多少条？ 起步可以很少（如 3 条）；生产要规划几十到几百条
有覆盖我们担心的失败模式吗？ 明确的"是"，附示例
有 commit 进版控吗？ 有 — 数据集是 PM 产物，不是 notebook cell
下次改 prompt 时会跑"同一份"数据集吗？ 会 — 同数据集、不同 prompt

产品场景

适合用 Claude 生成的情境

场景 为什么生成胜出
全新 AI 功能、没有先前数据集 从 0 到"可度量"的最快路径
探索新的输出格式 Claude 会提出 PM 可能没想到的任务变体
手工策划前先 bootstrap 省下数小时手写时间

适合手工构造的情境

场景 为什么手工胜出
法律、医疗、金融功能 数据集错误会变成产品错误
来自客服工单的真实客户边角情况 那些本来就是最好的测试案例
生产 regression suite 每一条都该对应一个已知失败模式

"成长中数据集"模式

给 PM 用的数据集演化心智模型：

  1. Bootstrap — 用 Haiku 生成约 3 条，先证明 eval pipeline 跑得起来。
  2. 扩展 — 让 Haiku 再生成 30 条以拓展广度；审核并保留好的。
  3. 策划 — 从真实客户投诉与边角情况加入手写案例。
  4. 锁定 — 把数据集 commit 进版控；每个新 prompt 版本都对它跑。
  5. 成长 — 每当生产出事故，把失败案例加进数据集，让它不再 regression。

第 5 步是让数据集变成产品与工程之间活的"质量合约"。

PM 常见错误

  1. 把数据集当成工程产物 — 它编码了产品的 definition-of-good，所以是 PM 交付物。
  2. 接受一份你从未读过的数据集 — 如果你没法扫过案例然后说"这就是真实使用"，那 eval 度量的就不是对的东西。
  3. 没把生产事故加进数据集 — 每一个客户可见的失败都该成为永久测试案例。
  4. 让工程师在两次跑之间重新生成数据集 — 这摧毁可比性并隐藏 regression。
  5. 没把 Haiku 的数据集刷新花费算进预算 — 便宜但不免费；算进功能的运营成本。

Key Insight

Eval 数据集是 PM 拥有的、对"这个功能的质量是什么"最清晰的答案。你纳入的每一条案例都是"我们保证产品会处理好这类输入"的明确承诺。CCA 考试把数据集工作归在 D3，凡是问"团队要如何为新 prompt bootstrap 测试数据"的题目，答案都指向用 Haiku 生成 + prefilling + 存盘重用。

CCA 考试相关性

  • D3（Evaluation & Iteration）：知道两种建数据集的做法、何时用哪一种，以及为什么 Haiku 是生成的默认。
  • D5（Enterprise Deployment）：存在磁盘上的版本化数据集是生产产物 — 它才让 prompt regression 可被检测。
  • 考题触发词："bootstrap eval dataset" → Haiku + prefilling + stop sequence + json.loads + 存盘。

Flashcards

Front Back
为什么 PM 该把 eval 数据集当成 PM 产物而不是工程产物？ 它编码了"这个功能什么叫好" — 那是产品判断，不是实现细节。
Eval 数据集的问卷类比？ 客户问卷 — 起草、试行、精修、锁定，每次产品改动都重新发放。
课程描述的两种建数据集做法是什么？ 手工构造（慢、高保真）和 Claude 生成（快、中保真）。
批量数据集生成该用哪个模型？ Haiku — 批量创意工作的又快又便宜选择。
当生产事故暴露新失败模式时 PM 该做什么？ 把失败案例加进 eval 数据集，下次迭代就能抓到 regression。
为什么数据集在 prompt 迭代间必须保持不变？ 这样 prompt 版本间的分数差异才能归因于 prompt 改动，而不是输入改动。
课程构建的最小可行第一版数据集是什么？ 三个 Claude 生成的任务（Python、JSON config、regex 各一）给 AWS code-assistant prompt。
哪个 CCA domain 涵盖数据集生成？ 主要：D3 Evaluation & Iteration；持久化的数据集也触及 D5 Enterprise Deployment 作为生产产物。
