一句話摘要

Eval 資料集編碼了你的 AI 功能「什麼叫好」 — 最快拿到可用第一版的方法是讓 Claude 幫你打草稿，然後把那個草稿當成一份 PM 擁有並持續迭代的活產出物。

心智模型：使用者問卷

把 eval 資料集想成客戶訪談問卷。PM 不會從白紙開始做使用者研究 — 他們會先擬問卷、測試幾個人、然後精修。資料集的做法完全一樣：

問卷概念 Eval 資料集概念
擬初版問卷題目 用 Haiku 產初版資料集
找幾個人試答 對生成的資料集跑一次 pilot eval
根據訊號精修題目 編輯資料集補強弱點
鎖定最終版 存成 dataset.json 並 commit 進版控
每季重新跑問卷 每次 prompt 改動都重跑 eval

資料集就是問卷。每一條 entry 都是一個「你產品要好好處理這種輸入」的決定。

AWS Code Assistant 範例

課程的 running example 是一個幫用戶寫 AWS 專屬 Python、JSON、regex 的 prompt。需求是乾淨輸出 — 沒有前言、沒有「這是你的程式碼」開場白。對 PM 來說，這個設定有三點值得注意：

  1. 輸出格式是產品需求，不是技術細節。 乾淨程式碼讓功能能在複製貼上工作流裡可用；雜亂程式碼會毀掉整個 use case。
  2. 資料集存在的意義，就是證明 prompt 能在非團隊手挑的輸入上遵守這個產品需求。
  3. 生成步驟產出的資料集能在客戶看到之前就以極低成本暴露格式失敗。

建立資料集的兩種做法

做法 速度 保真度 何時用
手工建 慢 高 — 每個案例都你控制 專業領域、受監管行業、高風險功能
Claude 生成 快 中 — 涵蓋取決於 prompt Bootstrapping、一般任務、早期功能

課程選 Claude 生成是有道理的：初始成本極低，而且你永遠可以事後手工編輯 — 編輯已生成的資料集比從零寫快得多。

成本槓桿： 課程叫工程師用 Haiku（不是 Sonnet）來生成。對 PM 來說，這直接對應到營運成本 — 批次資料集工作絕對不該用最貴的模型。

PM 決策框架

當團隊說他們產了一份 eval 資料集，走過這些問題：

問題 好答案長什麼樣
誰產的 — 人類還是 Claude？ 都可以，但答案會影響後續問題
如果是 Claude：用 Haiku 還是 Sonnet？ Haiku — 其他都是批次工作上的浪費
幾筆？ 起步可以很少（例如 3 筆）；production 要規劃數十到數百筆
有涵蓋我們擔心的失敗模式嗎？ 明確的「是」，附範例
有 commit 進版控嗎？ 有 — 資料集是 PM 產出物，不是 notebook cell
下次改 prompt 時會跑「同一份」資料集嗎？ 會 — 同資料集、不同 prompt

產品場景

適合用 Claude 生成的情境

場景 為什麼生成贏
全新 AI 功能、沒有先前資料集 從 0 到「可量測」的最快路徑
探索新的輸出格式 Claude 會提出 PM 可能沒想到的任務變化
手工策劃前先 bootstrap 省下數小時手寫時間

適合手工建的情境

場景 為什麼手工贏
法律、醫療、金融功能 資料集錯誤會變成產品錯誤
來自客服工單的真實客戶邊角案例 那些本來就是最好的測試案例
Production regression suite 每一條都該對應到一個已知失敗模式

「成長中資料集」模式

給 PM 用的資料集演化心智模型：

  1. Bootstrap — 用 Haiku 產 ~3 筆，先證明 eval pipeline 跑得起來。
  2. 擴展 — 叫 Haiku 再產 30 筆以拓展廣度；審核並保留好的。
  3. 策劃 — 從真實客戶抱怨與邊角案例加入手寫案例。
  4. 鎖定 — 把資料集 commit 進版控；每個新 prompt 版本都對它跑。
  5. 成長 — 每當 production 出事故，把失敗案例加進資料集，讓它不會再 regression。

第 5 步是讓資料集變成產品與工程之間活的「品質合約」。

PM 常見錯誤

  1. 把資料集當成工程產出物 — 它編碼了產品的 definition-of-good，所以是 PM 的交付物。
  2. 接受一份你從沒讀過的資料集 — 如果你沒辦法掃過案例然後說「這就是真實使用」，那 eval 量的就不是對的東西。
  3. 沒把 production 事故加進資料集 — 每一個客戶可見的失敗都該成為永久測試案例。
  4. 讓工程師在兩次跑之間重新生成資料集 — 這摧毀可比性並隱藏 regression。
  5. 沒把 Haiku 的資料集刷新花費算進預算 — 便宜但不免費；算進功能的營運成本。

Key Insight

Eval 資料集是 PM 擁有的、對「這個功能的品質是什麼」最清楚的答案。你納入的每一條案例都是「我們保證產品會處理好這類輸入」的明確承諾。CCA 考試把資料集工作歸在 D3，凡是問「團隊要如何為新 prompt bootstrap 測試資料」的題目，答案都指向用 Haiku 生成 + prefilling + 存檔重用。

CCA 考試相關性

  • D3（Evaluation & Iteration）：知道兩種建資料集的做法、何時用哪一種，以及為什麼 Haiku 是生成的預設。
  • D5（Enterprise Deployment）：存在磁碟上的版本化資料集是 production 產物 — 它才讓 prompt regression 可被偵測。
  • 考題觸發詞：「bootstrap eval dataset」→ Haiku + prefilling + stop sequence + json.loads + 存檔。

Eval 資料集的問卷比喻？ 客戶問卷 — 起草、試行、精修、鎖定，每次產品改動都重新發。

批次資料集生成該用哪個模型？ Haiku — 批次創意工作的快又便宜選擇。
當 production 事故暴露新失敗模式時 PM 該做什麼？ 把失敗案例加進 eval 資料集，下次迭代就能抓到 regression。
為什麼資料集在 prompt 迭代間必須保持不變？ 這樣 prompt 版本間的分數差異才能歸因於 prompt 改動，而不是輸入改動。
