Gotchas Around Hooks — PM Perspective

Item Details
Exam Coverage D3 — Claude Code Configuration & Workflows (20% of exam)
Task Statements 3.2 (custom commands & hooks), 1.5 (Agent SDK hooks)
Course Source claude-code-in-action / 05-hooks / Lesson 17 (text-only)

TL;DR

Hook scripts should use absolute file paths for security, but absolute paths are machine-specific and break when shared across a team. The solution is a template file (settings.example.json) with placeholders that an automated setup script converts into machine-specific absolute paths. PMs need to understand this because it affects onboarding, CI/CD configuration, and security compliance.

Why This Matters for PMs

This lesson addresses a deployment and governance challenge:

  1. Security audits — If your organization requires security reviews, hook commands with relative paths will be flagged
  2. Developer onboarding — New team members need to run a setup step before hooks work
  3. CI/CD pipelines — Hooks in automated environments need machine-specific configuration
  4. Team consistency — Without the template pattern, hooks may silently fail on some machines

Mental Model: Office Building Key Cards

Think of hook scripts as key card scanners at office building doors:

Aspect Key Card System Hook Configuration
Security requirement Scanner must connect to the exact security server (not any nearby server) Hook must point to the exact script file (absolute path)
Portability problem Each building has the security server at a different address Each developer machine has files at different paths
Solution Template card configuration + building-specific setup Template settings file + machine-specific init script
Onboarding New employee runs card activation at their building New developer runs npm run setup on their machine

💡 PM Takeaway

Absolute paths = higher security, but require an automated setup step. If your PRD mandates hooks, include "setup step required for new developer onboarding" in the deployment plan.

The Two Files Your Engineers Will See

After project setup, there are two settings files in the .claude directory:

File Who Creates It In Git? Contains Hooks?
settings.json Team lead, committed Yes General settings (no machine-specific paths)
settings.local.json Auto-generated by setup script No Hook commands with absolute paths

⚠️ PM Governance Note

If a compliance hook is only in settings.local.json, it depends on each developer running the setup script. Include this in your onboarding checklist and verify during code reviews.


Figure: The security vs. portability tradeoff — absolute paths are secure but machine-specific; template + init script solves both.

Security Concepts PMs Should Know

Two attack types that absolute paths prevent:

Attack Business Risk Prevention
Path interception Malicious code runs instead of the intended hook script Absolute path bypasses directory search
Binary planting Attacker places a fake script in the project directory Absolute path ignores local files with the same name

You do not need to understand the technical details, but you should know:
- Security audits will flag relative paths in hook configurations
- The CCA exam expects absolute paths as the correct answer for hook security questions

Onboarding Impact

When a new developer joins and does not run the setup script:

What Happens Impact Fix
settings.local.json does not exist No hooks are active Run npm run setup
Copy settings.example.json directly $PWD placeholders remain — hooks fail silently Run the init script properly
Manually type paths Error-prone, may introduce security risks Use the automated script

💡 PM Action Item

Add "run project setup script" to your team's onboarding documentation. Verify hooks are active by asking Claude to read a protected file.

Anti-Patterns (Exam Frequently Tested)

❌ Wrong Approach ✅ Correct Approach Why
Allow relative paths for convenience Require absolute paths for security Relative paths are flagged in security audits
Commit machine-specific settings to git Use template + init script pattern Machine-specific paths break on other machines
Skip setup step documentation Include setup in onboarding checklist New developers will have non-functional hooks
Assume hooks work after git clone Verify hooks are active after setup Generated files are not in git

Practice Questions

Q1: Developer Productivity Scenario (S4)

Your team has deployed a PreToolUse hook to prevent Claude from accessing credential files. A new developer joins, clones the repo, and starts using Claude Code. Two days later, they discover Claude was able to read credential files on their machine. What is the most likely cause?

  • A. Claude Code has a bug that bypasses hooks on certain operating systems
  • B. The new developer did not run the project setup script, so settings.local.json was never generated
  • C. The hook was configured at the global level and was not included in the project
  • D. PreToolUse hooks only work for the Read tool, not Grep
Answer **B** — The `settings.local.json` file is gitignored and must be generated by the setup script. Without running setup, no hooks are active. - A: Hooks are platform-independent - C: If other team members have working hooks from the same repo, it is a project-level configuration - D: PreToolUse hooks work with any tool specified in the matcher **PM Takeaway**: Hook activation is part of the development environment setup. Treat it like database migration or dependency installation — it must be in the onboarding checklist.

Q2: CI/CD Integration Scenario (S5)

Your security team requires all hook scripts to use absolute paths. Your CI pipeline runs on dynamically provisioned runners where the workspace directory changes each run. How should you configure hooks for CI?

  • A. Use relative paths in CI since security is less important in automated environments
  • B. Add a CI pipeline step that runs the init script to generate settings.local.json with the current workspace's absolute paths
  • C. Hardcode the most common CI workspace path in settings.json
  • D. Disable hooks in CI and rely on other security measures
Answer **B** — The init script pattern works in CI too. A setup step generates machine-specific absolute paths from the template. - A: Security requirements apply equally in CI — automated environments are also attack surfaces - C: Dynamic runners have different paths each time - D: If hooks enforce compliance, disabling them in CI creates a gap **PM Takeaway**: CI/CD environments need the same security posture as developer machines. Include the setup step in your CI pipeline configuration.

Q3: Customer Support Scenario (S1)

During a security audit of your AI customer support system, the auditor notes that hook commands use relative paths. The engineering team argues that absolute paths would make settings files harder to share. What is the recommended solution?

  • A. Accept the security risk since the hooks are only used internally
  • B. Use a settings.example.json template with $PWD placeholders and an automated init script
  • C. Remove hooks and rely on system prompt instructions instead
  • D. Use relative paths but add a CLAUDE.md instruction to not run malicious scripts
Answer **B** — The template + init script pattern satisfies both requirements: absolute paths for security and portability for team sharing. - A: Security risks should never be accepted when a solution exists - C: Removing hooks eliminates deterministic enforcement — a major regression - D: CLAUDE.md instructions cannot prevent path interception attacks **PM Takeaway**: When engineers present a security-convenience trade-off, look for solutions that satisfy both. The template pattern is the standard solution for this specific trade-off.
