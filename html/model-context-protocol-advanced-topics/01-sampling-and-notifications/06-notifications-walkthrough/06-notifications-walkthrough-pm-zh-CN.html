一句话摘要

MCP 通知让服务器工具在长时间操作期间向客户端传达实时状态（日志）和进度更新——就像外卖员在完成你的订单时发送实时 GPS 更新和状态消息。

PM 为何需要了解通知

通知解决了一个根本的用户体验问题：长时间 AI 操作期间会发生什么？ 没有通知，用户只能盯着转圈圈。有了通知，你可以构建：

  • 显示任务完成百分比的进度条
  • 解释 AI 目前正在做什么的状态消息
  • 操作遇到问题时的警告指示器
  • 用于故障排除的调试日志

心智模型：快递追踪

配送阶段 MCP 通知 用户看到什么
「订单已收到，准备中...」 ctx.info("Preparing to process...") UI 中的状态消息
「包裹已完成 40%」 ctx.report_progress(40, 100) 进度条在 40%
「路线上有交通延误」 ctx.warning("Processing slower than expected") 警告横幅
「已送达！」 工具返回结果 完成状态

两种通知类型

类型 传达的内容 PM 使用场景
日志记录 不同严重等级的文本状态消息（info、warning、debug、error） 构建活动动态、状态指示器、错误警报
进度报告 数值完成度（例如 20/100、80/100） 构建进度条、预估剩余时间

架构：各组件如何连接

这对撰写需求的 PM 很重要，因为它影响工程师如何组织代码：

组件 职责 PM 影响
服务器工具 在执行期间发送日志和进度 服务器团队决定要报告什么
日志回调（在会话上） 处理整个连接的所有日志消息 会话级别功能——始终启用
进度回调（在工具调用上） 处理特定工具调用的进度 每次操作的功能——每次都必须明确附加

关键要点：如果你的需求写「显示文件转换的进度」，进度回调必须在调用该特定工具时明确传入。它不是自动的。

产品场景

场景 1：视频处理仪表板

需求：在视频转换时向用户显示进度条。

实现方式：服务器上的视频转换工具在里程碑处调用 ctx.report_progress()（编码开始、完成 50%、最终处理）。客户端通过进度回调接收这些更新并更新 UI。

场景 2：多步骤研究智能体

需求：向用户显示智能体在研究任务每个阶段正在做什么。

实现方式：服务器工具调用 ctx.info("Searching academic databases...")、ctx.info("Found 12 relevant papers...")、ctx.info("Generating summary...")。每条消息都出现在客户端的活动动态中。

场景 3：错误监控

需求：当工具执行遇到问题时警告运维团队。

实现方式：服务器工具在性能降低时调用 ctx.warning() 或 ctx.error()。日志回调将警告导向监控仪表板。

通知 vs 其他方法

方法 何时使用 何时不使用
MCP 通知 长时间工具执行期间的实时状态 瞬间完成的快速操作
工具返回值 仅最终结果 用户需要中间反馈时
客户端轮询 服务器无法推送更新时 服务器支持通知时（MCP 支持）
WebSocket 流 MCP 之外的实时双向通信 已使用 MCP 时（改用通知）

CCA 考试相关性

  • D2 测试对回调连接位置的理解（日志在会话上，进度在工具调用上）
  • 通知基本要素是发射即忘——服务器不等待客户端确认
  • 考题可能提出一个用户体验场景，询问哪种通知类型适合（日志用于消息，进度用于数值跟踪）
  • Context 对象会自动注入到工具函数中——不是手动构建的

记忆卡

# 问题 答案
1 MCP 支持哪两种类型的通知？ 日志记录（文本消息）和进度报告（数值完成度更新）
2 客户端代码中日志回调连接在哪里？ ClientSession 构造函数上
3 进度回调连接在哪里？ 特定的 call_tool() 调用上
4 为什么它们连接在不同的级别？ 日志记录适用于整个会话；进度针对每次工具调用
5 MCP 通知是阻塞式还是发射即忘？ 发射即忘——服务器不等待客户端
6 有哪些可用的日志严重等级？ info、warning、debug、error
7 服务器如何访问通知方法？ 通过 Context 对象，自动作为工具函数的最后一个参数提供
8 通知解决了什么产品问题？ 长时间操作期间的用户体验——用实时状态取代空白转圈圈
