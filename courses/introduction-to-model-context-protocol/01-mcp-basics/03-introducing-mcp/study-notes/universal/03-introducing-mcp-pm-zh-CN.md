## 一句话摘要

MCP 是一个通用转接标准，让 AI 助手能即插即用地连接任何外部服务，无需为每个服务做定制化集成。

---

## 商业问题：集成蔓延

想象你在管理一个产品团队，正在打造面向客户的 AI 助手。助手需要访问 GitHub 获取工程数据、Salesforce 获取客户资料、Slack 用于内部沟通。没有标准协议的话，你的工程团队必须为每个服务构建和维护独立的集成——就像为办公室里的每个电器打造专用电源转接器。

这造成三个不断累积的问题：

1. **开发成本** — 每个新服务都需要数周的定制化集成工作
2. **维护负担** — 任何服务更新 API 时，你的团队就得更新集成
3. **质量风险** — 每个手工打造的集成都是另一个 bug 和安全漏洞的攻击面

这正是硬件世界在 USB 出现之前的状况。每台打印机、键盘、鼠标都有自己的专有连接器。USB 标准化了接口，改变了整个产业。

> **PM Takeaway**
> 评估是否为产品采用 MCP 时，关键问题不是"我们能自己建集成吗？"而是"我们该把工程人力花在集成管线上，还是产品功能上？"

---

## MCP 如何改变游戏规则

MCP 引入了一个所有服务都能实现的标准"插头"。你的团队不用建定制化集成，而是连接到已经知道如何与各服务沟通的预建 MCP server。

想象一下两者的差异：

- **MCP 之前**：聘请一位英翻西语的翻译、一位英翻法语的翻译、一位英翻日语的翻译——每位都为你的特定需求定制化培训
- **MCP 之后**：使用通用翻译服务，翻译人员遵循标准协议，你可以随时替换，不需重新培训任何人

MCP 架构有两个关键角色：

**MCP Client** — 你的产品（用户交互的对象）。它知道如何请求 tools 并使用它们，但不需要知道每个外部服务内部如何运作。

**MCP Server** — 针对特定服务（GitHub、Salesforce 等）的专用连接器。它处理所有服务特定的复杂性，呈现干净、标准化的接口。

> **PM Takeaway**
> MCP 是"买 vs 建"的加速器。生态系统中每存在一个 MCP server，你的团队就省下数周的集成工作。ROI 随你的 AI 产品需要访问的服务数量而扩大。

---

## MCP 生态系统：谁建什么

MCP 最重要的特征之一是其开放生态系统。MCP server 可由以下人员构建：

- **服务提供者**（第一方）：像 AWS 或 Stripe 这样的公司为其平台发布官方 MCP server
- **社区开发者**（第三方）：开源贡献者为热门服务构建和维护 server
- **你自己的团队**（内部）：针对没有公开 MCP server 覆盖的专有内部工具和数据库

这映射了 App Store 模式。Apple 建平台（MCP 协议），部分 app 来自大公司（官方 MCP server），任何人都能发布自己的（自定义 server）。

## MCP vs. Tool Use：常见混淆

利益相关者甚至部分工程师会混淆这两个概念。以下是清楚的区分：

**Tool use** 是能力——Claude 可以调用函数、获取数据、执行动作。这无论有没有 MCP 都存在。

**MCP** 是交付机制——它提供 tool 定义和执行基础设施，让你的团队不需从零开始构建。

打个比方：tool use 就像打电话的能力。MCP 就像电话簿和总机服务。你可以不用电话簿打电话（自己查号码手动拨号），但电话簿让整个过程快得多、可靠得多。

> **PM Takeaway**
> 在产品讨论中，把 MCP 定位为工具集成的"how"，而非"what"。"what"是 Claude 使用 tools 的能力。MCP 只是让工具集成在交付上大幅更便宜、更快速。

## 对产品决策的战略影响

规划产品路线图时，MCP 影响几个关键决策：

**上市时间**：当 MCP server 已存在时，新增集成从数周降到数天（或数小时）。

**买 vs 建**：对标准服务，使用现有 MCP server 几乎永远是正确选择。自定义 MCP server 只对专有内部系统有意义。

**厂商锁定**：MCP 是开放协议，非 Anthropic 专有。这降低对单一 AI 供应商的依赖。

**可扩展性**：你的产品可以支持数十个集成，而不需要等比例增加工程人员。
