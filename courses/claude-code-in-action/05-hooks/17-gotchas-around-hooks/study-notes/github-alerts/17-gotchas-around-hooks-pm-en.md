---

## TL;DR

Hook scripts should use absolute file paths for security, but absolute paths are machine-specific and break when shared. The solution is a template file with placeholders that an automated setup script converts into machine-specific paths. PMs need to understand this because it affects onboarding, CI/CD, and security compliance.

---

## Why This Matters for PMs

1. **Security audits** — Relative paths in hook commands will be flagged
2. **Developer onboarding** — New members need to run a setup step before hooks work
3. **CI/CD pipelines** — Hooks in automated environments need machine-specific configuration
4. **Team consistency** — Without the template pattern, hooks may silently fail on some machines

---

## Mental Model: Office Building Key Cards

| Aspect | Key Card System | Hook Configuration |
|--------|----------------|-------------------|
| Security requirement | Scanner must connect to exact security server | Hook must point to exact script file (absolute path) |
| Portability problem | Each building has different server address | Each machine has files at different paths |
| Solution | Template card config + building-specific setup | Template settings + machine-specific init script |

> [!TIP]
> **PM Takeaway**
>
> Absolute paths = higher security, but require an automated setup step. If your PRD mandates hooks, include "setup step required for onboarding" in the deployment plan.

---

## The Two Files

| File | Who Creates It | In Git? | Contains Hooks? |
|------|---------------|---------|----------------|
| `settings.json` | Team lead, committed | **Yes** | General settings (no machine-specific paths) |
| `settings.local.json` | Auto-generated by setup script | **No** | Hook commands with absolute paths |

> [!WARNING]
> **PM Governance Note**
>
> If a compliance hook is only in `settings.local.json`, it depends on each developer running the setup script. Include this in your onboarding checklist.

---

## Security Concepts PMs Should Know

| Attack | Business Risk | Prevention |
|--------|--------------|-----------|
| **Path interception** | Malicious code runs instead of the intended hook | Absolute path bypasses directory search |
| **Binary planting** | Attacker places fake script in project directory | Absolute path ignores local same-name files |

> [!IMPORTANT]
> Security audits will flag relative paths. The CCA exam expects absolute paths as the correct answer.

---

## Onboarding Impact

| What Happens | Impact | Fix |
|-------------|--------|-----|
| `settings.local.json` missing | No hooks active | Run `npm run setup` |
| Direct copy of template | `$PWD` placeholders remain — hooks fail silently | Run init script properly |

> [!TIP]
> **PM Action Item**
>
> Add "run project setup script" to your team's onboarding documentation. Verify hooks are active by asking Claude to read a protected file.

---

## Anti-Patterns (Exam Frequently Tested)

| ❌ Wrong Approach | ✅ Correct Approach | Why |
|-------------------|---------------------|-----|
| Allow relative paths for convenience | Require absolute paths for security | Security audits flag relative paths |
| Commit machine-specific settings | Use template + init script pattern | Machine paths break on other machines |
| Skip setup documentation | Include setup in onboarding checklist | New devs will have broken hooks |
| Assume hooks work after git clone | Verify hooks are active after setup | Generated files are not in git |

## Practice Questions

### Q1: Developer Productivity Scenario (S4)

A new developer joins and Claude can read credential files on their machine, despite the team having a PreToolUse hook. Most likely cause?

- A. Claude Code bug on their OS
- B. They did not run the setup script — `settings.local.json` was never generated
- C. Hook configured at global level only
- D. PreToolUse only works for Read, not Grep

<details><summary>Answer</summary>

**B** — `settings.local.json` is gitignored and must be generated by the setup script.

> [!IMPORTANT]
> **PM Takeaway**: Hook activation is part of environment setup — treat it like database migration in your onboarding checklist.
</details>

### Q2: CI/CD Integration Scenario (S5)

Security team requires absolute paths for hooks. CI runs on dynamic runners. How to configure?

- A. Use relative paths in CI since security is less important
- B. Add a CI step that runs the init script to generate `settings.local.json` with workspace absolute paths
- C. Hardcode the most common CI path
- D. Disable hooks in CI

<details><summary>Answer</summary>

**B** — The init script pattern works in CI too.

> [!IMPORTANT]
> **PM Takeaway**: CI/CD environments need the same security posture as developer machines.
</details>

### Q3: Customer Support Scenario (S1)

Security audit flags hook commands using relative paths. Engineers say absolute paths make sharing harder. Recommended solution?

- A. Accept the security risk
- B. Use `settings.example.json` with `$PWD` placeholders and an automated init script
- C. Remove hooks, use system prompt instead
- D. Use relative paths with a CLAUDE.md warning

<details><summary>Answer</summary>

**B** — Template + init script satisfies both security (absolute paths) and portability (team sharing).

> [!IMPORTANT]
> **PM Takeaway**: When engineers present a security-convenience trade-off, look for solutions that satisfy both. The template pattern is the standard answer.
</details>
