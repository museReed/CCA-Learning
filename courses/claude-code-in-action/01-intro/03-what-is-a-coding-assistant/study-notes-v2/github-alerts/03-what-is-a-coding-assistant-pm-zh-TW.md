## One-Liner / TL;DR

Coding assistant 就像一個能思考問題又能動手做事的資深承包商 — 不只是站在旁邊給建議而已。

## Core Concepts

### Coding Assistant 究竟是什麼

Coding assistant 不僅僅是一個寫程式碼的工具。把它想像成一個完整的問題解決系統：它理解你的需求、弄清楚需要什麼資訊、制定計畫，然後執行。「助手」這個角色是關鍵 — 它代你行動，就像一個能思考又能動手的真人助理。

### Coding Assistant 的運作方式 — 三步驟循環

當你給 coding assistant 一個任務（例如「根據這個錯誤訊息修復 bug」），它會遵循一個可重複的流程：

| 步驟 | 做什麼 | 職場比喻 |
|------|--------|---------|
| 1 | **蒐集上下文** — 理解錯誤、找到相關檔案、閱讀相關程式碼 | 像新團隊成員在處理問題前先翻閱 Slack 訊息和文件 |
| 2 | **制定計畫** — 決定修復問題的最佳方法 | 像資深工程師在寫程式碼前先在白板上勾勒解決方案 |
| 3 | **採取行動** — 透過編輯檔案、執行命令來實施修復 | 像工程師實際打開程式碼庫並進行修改 |

> [!IMPORTANT]
> 步驟 1 和 3 需要**與外部世界互動** — 讀取檔案、執行命令。這就是魔法（和挑戰）所在。就像給你的助手一把瑞士刀 — 每個工具讓它能以不同方式與真實世界互動。

### Tool Use 的挑戰

這是根本的限制：語言模型（AI「大腦」）只能處理文字並返回文字。它無法：
- 開啟和讀取你電腦上的檔案
- 執行終端機命令
- 寫入或編輯程式碼檔案
- 瀏覽網路

如果你問一個普通的語言模型「main.go 裡有什麼？」，它會告訴你它無法存取你的檔案。它只能處理你給它的文字 — 就像一個只能透過書信溝通的專家顧問。

### Tool Use 的運作方式 — 完整流程

這就是 coding assistant 如何彌補差距的。課程中的 `ReadFile: main.go` 範例完整展示了發生什麼事：

| 步驟 | 發生什麼事 | 白話解釋 |
|------|-----------|---------|
| 1 | **你提問**：「main.go 檔案裡寫了什麼程式碼？」 | 你向 coding assistant 傳送一個問題 |
| 2 | **Coding assistant** 在你的請求後附加工具指令 | 助手告訴 AI 大腦：「嘿，你有這些工具可以用 — 這是使用方法」 |
| 3 | **語言模型回應**：`ReadFile: main.go` | AI 大腦說：「我需要讀那個檔案 — 請對 main.go 使用 ReadFile 工具」 |
| 4 | **Coding assistant** 讀取實際檔案並將內容傳回 | 助手去讀取真正的檔案，然後把內容交回給 AI 大腦 |
| 5 | **語言模型**根據檔案內容提供最終回答 | 現在 AI 大腦能給你真正的答案，因為它看到了實際的程式碼 |

> [!NOTE]
> 這叫做**「tool use」** — 業界的標準術語。每個 coding assistant 都這樣運作，但有些 AI 模型做得比其他的好得多。

### 為什麼 Claude 的 Tool Use 很重要

Claude（AI 模型：Opus、Sonnet、Haiku）在 tool use 方面特別強。三個具體好處：

| 好處 | 對你的團隊意味著什麼 |
|------|-------------------|
| **處理更困難的任務** | Claude 能以創造性、出乎意料的方式組合工具，還能使用從未遇過的工具 — 像一個不需要培訓就能搞懂新軟體的機靈員工 |
| **可擴展平台** | 容易新增功能 — Claude 無需重新訓練即可適應。如果團隊需要新的整合，直接加工具就好 |
| **更好的安全性** | 不需要預先掃描你的整個程式碼庫或發送到外部伺服器。Claude 按需讀取檔案 — 你的程式碼留在本地 |

## Instructor Tips

> [!TIP]
> 「Coding assistant 不僅僅是一個寫程式碼的工具」— 講師強調，理解底層發生什麼事能幫助你更有效地使用這些工具，並對何時及如何部署它們做出更好的決策。

> [!TIP]
> 逐步示範清楚展示 AI 模型從未真正「讀取」檔案 — coding assistant（模型外面的軟體包裝層）負責做真實世界的工作。模型只是傳達它需要什麼。

## Key Takeaways

1. Coding assistant 使用語言模型（AI 大腦）來完成複雜的程式設計任務
2. 語言模型需要工具才能做真實世界的工作 — 它們無法自己讀取檔案或執行命令
3. 並非所有語言模型的 tool use 能力都一樣好 — 這是關鍵的差異化因素
4. Claude 強大的 tool use 能力帶來更好的安全性（程式碼留在本地）、容易客製化（新增工具）和處理更困難問題的能力

## Familiar Analogies

- **行政助理比喻** — 沒有工具的語言模型就像一個被鎖在房間裡、沒有電話、電腦和檔案的天才助理。如果你帶資料給他，他能提供很好的建議，但他無法自己查任何東西。Tool use 就是把他的電話和電腦還給他。
- **食譜 vs. 廚房** — 知道食譜（模型的知識）但沒有廚房（工具）是沒用的。Coding assistant 就是讓廚師（模型）能真正下廚的廚房。
- **App 整合** — 就像 Slack 加上整合功能（Google Drive、Jira、GitHub）後變得更強大，語言模型加上工具也是。每個工具就像一個整合，讓它能做新的事情。

## CCA Exam Connection

> [!TIP]
> **考試提示**：本單元教授每個 coding assistant 背後的基礎架構。需要記住的關鍵概念：
> - 三步驟循環：蒐集上下文、制定計畫、採取行動
> - 語言模型無法直接存取檔案或執行命令 — 需要 tool use
> - 五步驟 tool use 流程（特別是：每一步驟誰做什麼）
> - Claude 的三大優勢：更困難的任務、可擴展性、安全性
> - 語言模型（思考）和 coding assistant（行動）的區別

## Anti-Patterns

| 常見誤解 | 為什麼是錯的 | 實際情況 |
|---------|-------------|---------|
| 「AI 直接讀取我的檔案」 | 語言模型只處理文字 — 沒有檔案系統存取權限 | Coding assistant 讀取檔案並將文字內容傳遞給模型 |
| 「所有 AI 程式碼工具基本上都一樣」 | Tool use 品質在不同模型間差異很大 | Claude 被特別指出在 tool use 方面特別強 |
| 「我的程式碼會被上傳到雲端」 | 這是假設基於索引的架構 | Claude 的 tool-use 方法按需讀取檔案 — 你的程式碼留在你的機器上 |
| 「Coding assistant 只是進階版的自動完成」 | 自動完成是單一、狹隘的動作 | Coding assistant 執行完整循環：理解問題、計畫、執行 — 可能經過多個回合 |

## Practice Questions

**Q1.** 在 tool use 互動中，語言模型回應了 `ReadFile: main.go`。接下來會發生什麼？

- A) 語言模型直接從電腦的檔案系統開啟檔案
- B) Coding assistant 攔截回應，從磁碟讀取檔案，並將內容傳回模型
- C) 使用者必須手動將檔案內容複製貼上到聊天中
- D) 模型從雲端儲存庫下載檔案

> [!NOTE]
> **答案：B。** 語言模型無法存取檔案系統。Coding assistant（軟體包裝層）攔截結構化的工具呼叫，讀取實際檔案，並以文字形式將內容傳回給模型處理。

**Q2.** 一位產品經理正在評估 coding assistant。以下哪一項是 Claude 強大 tool use 能力的明確好處？

- A) 它處理程式碼的速度比其他語言模型更快
- B) 它需要預先索引整個程式碼庫才能達到最佳效果
- C) 它能以創造性的方式組合工具來處理更困難的任務
- D) 它只能使用它被特別訓練過的工具

> [!NOTE]
> **答案：C。** Claude 能以有趣且出乎意料的方式組合工具，甚至能使用從未見過的工具。速度（A）未被提及，預先索引（B）與課程所述相反，而（D）與可擴展性好處矛盾。
