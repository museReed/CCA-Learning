# PART 1: Official Course Content

> All content in this section comes directly from official course materials.

## One-Liner / TL;DR

CLAUDE.md is your team's onboarding document for the AI assistant — `/init` generates it, a three-level hierarchy enforces standards across the organization, and `@` file references control what the AI always knows about your project.

## Core Concepts

### Context Management Principle

Think of Claude Code like a new contractor joining your project. If you dump every document in the company on their desk, they will be overwhelmed and perform poorly. If you give them just the key architecture docs, coding standards, and point them to the right files, they ramp up quickly.

The course explicitly states: too much irrelevant context decreases Claude Code's effectiveness. A typical project has dozens or hundreds of files. The ideal is giving Claude just enough to complete the task — no more, no less. This directly impacts developer productivity and AI output quality.

### The /init Command — Automated Project Onboarding

When Claude Code is run in a new project for the first time, the `/init` command is like running an automated onboarding session. Claude:

1. **Reads the entire codebase** — understands project purpose, architecture, and patterns
2. **Identifies what matters** — key commands, critical files, project structure
3. **Creates an onboarding document** — generates a CLAUDE.md file summarizing its findings

The developer approves the file creation (Enter to accept, Shift+Tab for auto-accept mode). This is a one-time setup per project.

**PM Takeaway:** `/init` eliminates the "AI doesn't know our project" problem. One command, and every developer's AI assistant understands the project architecture.

### The CLAUDE.md File — Persistent Team Configuration

CLAUDE.md serves two purposes:

1. **Project knowledge base for the AI** — architecture, commands, coding style (like an internal wiki that the AI actually reads)
2. **Custom behavioral directives** — rules the AI follows on every interaction

The contents are included in every request to Claude — it is a persistent system prompt. Think of it as a configuration file that shapes all AI behavior on the project.

### CLAUDE.md File Locations — Three-Tier Policy System

Claude Code recognizes CLAUDE.md files at three levels, functioning like a corporate policy hierarchy:

| Level | File | Shared? | PM Analogy |
|-------|------|---------|------------|
| Project | `./CLAUDE.md` | Yes — committed to source control, shared with team | **Department policy** — applies to everyone on the project. Generated by `/init`, version-controlled in Git. |
| Local | `./CLAUDE.local.md` | No — not committed to source control | **Individual exception** — personal instructions for this project. Not visible to other team members. |
| Global | `~/.claude/CLAUDE.md` | No — machine-specific | **Corporate-wide policy** — applies to all projects on this developer's machine. Personal cross-project preferences. |

> **For team standardization:** Project CLAUDE.md is the lever. It is committed to source control, which means every developer who clones the repo inherits the same AI configuration. This is how you enforce consistent coding standards across a team without relying on individual discipline.

### Custom Instructions with # Memory Mode — Updating the Team Wiki

The `#` command in Claude Code enters "memory mode" — a shortcut for updating CLAUDE.md files without opening them manually. The developer types an instruction, selects which CLAUDE.md file to add it to, and Claude merges it intelligently.

**Example:** A tech lead notices Claude writes too many code comments. They type:
```
> # Don't write comments so often
```
This gets merged into the project CLAUDE.md. Now every developer on the team benefits from this instruction.

### File Mentions with @ — Controlling AI Focus

The `@` syntax includes specific file contents in requests to Claude. Two usage patterns exist:

**In chat (ad-hoc, one-time):**
```
> How does the auth system work? @auth
```
The file contents are included only in that request. Good for exploration.

**In CLAUDE.md (persistent, every request):**
```markdown
The database schema is defined in @prisma/schema.prisma.
```
The referenced file is automatically included in every request. Claude can answer questions about data structure instantly without searching.

> **Context window warning** — Every `@` reference in CLAUDE.md consumes context window space on every request. This is a budget trade-off: more persistent context = less room for Claude to reason. Only put critical, cross-cutting files here.

## Instructor Tips

- `/init` is a one-time investment that pays off on every subsequent interaction
- CLAUDE.md is the single most impactful configuration for controlling AI behavior across a team
- The `#` shortcut lets tech leads update team-wide AI behavior without opening files
- Be strategic about `@` references — each one is a permanent context cost
- Review CLAUDE.md periodically to remove stale or low-value instructions

## Key Takeaways

1. Too much irrelevant context degrades AI performance — less is more
2. `/init` generates CLAUDE.md by analyzing the entire codebase — automated project onboarding for the AI
3. CLAUDE.md is included in every request — it is the team's AI configuration file
4. Three levels: project (team-wide, version-controlled), local (personal), global (all projects)
5. `#` memory mode updates CLAUDE.md without manual editing — fast behavioral adjustment
6. `@` in chat = one-time context; `@` in CLAUDE.md = persistent context on every request
7. Context window is a finite budget — optimize `@` references like you optimize database queries

## Familiar Analogies

| Concept | PM Analogy | Why It Fits |
|---------|-----------|-------------|
| CLAUDE.md | Team onboarding document for the AI assistant | Tells the AI everything it needs to know about the project, loaded every time |
| CLAUDE.md hierarchy | Corporate policy levels: company-wide < department < individual exceptions | More specific overrides more general; local preferences override team defaults |
| `/init` command | Running a new employee through onboarding — read all docs, understand the org | One-time setup that pays dividends on every future interaction |
| `@` in CLAUDE.md | Standing agenda items in a recurring meeting | Always discussed, always present — consumes time but ensures coverage |
| `@` in chat | Ad-hoc agenda items brought up in one meeting | One-time, context-specific — does not consume recurring time |
| `#` memory command | Updating the team wiki or knowledge base | Persistent change that survives personnel turnover |
| Context window budget | Meeting time budget — 60 minutes total | Loading too many standing items leaves no time for new discussion |

## CCA Exam Connection

As a PM, focus on these exam angles:

**Team standardization (Task 3.1)** — Project CLAUDE.md committed to source control = consistent AI behavior across all developers. This is your policy enforcement mechanism.

**Onboarding efficiency (Task 5.1)** — New developers inherit project context via CLAUDE.md automatically. No manual AI training needed.

**Performance management (Task 5.4)** — If developers report "Claude is slow/inaccurate," check for context overload in CLAUDE.md. Too many `@` references is the most common culprit.

**Decision rule:** "Would a new team member need this?" If yes, it goes in project CLAUDE.md. If personal, use local or global.

## Anti-Patterns

| Anti-Pattern | Why It Fails | Correct Approach |
|-------------|-------------|-----------------|
| No CLAUDE.md in the repo | Every developer's AI starts from zero each session | Run `/init`, commit CLAUDE.md, maintain it |
| Using CLAUDE.local.md for team standards | Local files are gitignored — no team visibility | Use project CLAUDE.md for shared standards |
| Overloading CLAUDE.md with `@` references | Context window saturation degrades performance | Audit references quarterly; keep only cross-cutting files |
| Each developer configures their own AI rules | Creates inconsistency and configuration drift | Centralize in project CLAUDE.md via source control |
| Never reviewing CLAUDE.md after initial generation | Stale instructions mislead the AI | Treat CLAUDE.md like documentation — review and update regularly |
| Putting personal style preferences in project CLAUDE.md | Imposes one person's preferences on the whole team | Use CLAUDE.local.md or ~/.claude/CLAUDE.md for personal prefs |

## Practice Questions

**Q1.** Your CTO asks: "How do we ensure all 30 developers using Claude Code follow our coding standards?" Which approach is correct?

- A) Send an email asking each developer to configure their settings
- B) Add coding standards to the project CLAUDE.md and commit it to the repository
- C) Create a CLAUDE.local.md template and ask each developer to copy it
- D) Use the Anthropic dashboard to set organization-wide rules

> **Answer: B.** Project CLAUDE.md is committed to source control and automatically applied to every developer who clones the repo. No manual setup, no drift. Option C uses the wrong file type (local is gitignored). Option D does not exist.

**Q2.** A developer reports Claude Code responses are getting slower and less accurate since last week. Investigation reveals they added 12 `@` file references to CLAUDE.md. What do you recommend?

- A) Upgrade to a higher API tier for more context window
- B) Audit `@` references — keep only cross-cutting files, move the rest to interactive `@` mentions
- C) Remove CLAUDE.md entirely and start over
- D) Switch to a simpler project structure

> **Answer: B.** The course explicitly teaches that too much context degrades performance. Audit and optimize — keep schema files and API contracts, remove task-specific references. This is proportionate and targeted.

**Q3.** A new developer joins your team. They have never used Claude Code. What is the fastest path to productive AI-assisted development?

- A) Provide a 2-hour training on prompt engineering
- B) Have them install Claude Code and pull the repo — the committed CLAUDE.md provides project context automatically
- C) Ask them to run `/init` to generate a fresh CLAUDE.md
- D) Share your personal CLAUDE.local.md with them

> **Answer: B.** If the team maintains a committed CLAUDE.md, the new developer inherits all project context automatically. Option C would overwrite the team's existing CLAUDE.md. Option D shares personal, gitignored settings.

**Q4.** Where should a developer put personal coding preferences (like "always use dark theme examples") that should not affect the team?

- A) Project CLAUDE.md
- B) CLAUDE.local.md or ~/.claude/CLAUDE.md
- C) A comment in the codebase
- D) An environment variable

> **Answer: B.** CLAUDE.local.md is for per-project personal preferences (gitignored). ~/.claude/CLAUDE.md is for cross-project personal preferences (machine-specific). Neither is shared with the team.
