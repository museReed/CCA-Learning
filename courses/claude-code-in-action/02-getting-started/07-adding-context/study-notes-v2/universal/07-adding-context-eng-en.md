## One-Liner / TL;DR

Context management is the single most impactful skill in Claude Code — `/init` generates CLAUDE.md, the three-level CLAUDE.md hierarchy provides persistent instructions, `#` memory mode updates them, and `@` file mentions inject specific file contents on demand.

## Core Concepts

### Context Management Principle

Too much irrelevant context decreases Claude Code's effectiveness. Inside a typical project there might be dozens or hundreds of files, each with a tremendous amount of information. Whenever you ask Claude a question or give it a task, there is an ideal amount of information — just enough to understand how to answer or complete the task. As soon as you start adding irrelevant information, effectiveness decreases.

It is important to guide Claude towards relevant files and documentation. Claude Code can work without handholding, but you get the best results with a little bit of guidance.

### The /init Command

When you first open Claude Code in a new project, run `/init`. Claude will:

1. **Analyze the entire codebase** — project purpose, general architecture, coding patterns and structure
2. **Identify key elements** — relevant commands, critical files, project structure
3. **Generate a CLAUDE.md file** — summarize its findings and place them into this file

When Claude tries to create the file, it asks for permission:
- **Enter** — approve the file write
- **Shift+Tab** — allow Claude Code to freely write files in your project (auto-accept mode)

### The CLAUDE.md File

The CLAUDE.md file has two purposes:

1. **Guides Claude through the codebase** — helps Claude better understand your codebase so it can find relevant code more quickly (commands, architecture, style)
2. **Custom directions** — serves as a location where you can give Claude general guidance

The contents of this file are included in every request you make to Claude — it functions as a persistent system prompt.

### CLAUDE.md File Locations

Claude Code recognizes CLAUDE.md files at three levels:

| Level | File | Shared? | Description |
|-------|------|---------|-------------|
| Project | `./CLAUDE.md` | Yes — committed to source control | Generated by `/init`. Shared with other engineers via Git. Contains project-specific directions for Claude. |
| Local | `./CLAUDE.local.md` | No — not committed | Personal instructions that you want Claude to follow just for you. Not shared with any other engineers. |
| Global | `~/.claude/CLAUDE.md` | No — machine-specific | Instructions applied to all projects that you run locally on this machine. |

> The instructor encourages opening the generated CLAUDE.md file and reviewing its contents. Since it is included in every request, understanding what it contains helps you optimize context.

### Custom Instructions with # Memory Mode

To update CLAUDE.md without manually editing the file, use the `#` command in Claude Code. This puts you in "memory mode," which allows you to edit one of your CLAUDE.md files intelligently.

**Example:**
```
> # Don't write comments so often
```

After entering the instruction, you specify which CLAUDE.md file to add it to (project, local, or global). Claude then merges the instruction into that file intelligently — it does not blindly append.

### File Mentions with @

Use `@` followed by a file path to include file contents in your request. This is a technique for pointing Claude in a specific direction.

**In chat (one-time context):**
```
> How does the auth system work? @auth
```
When you mention a file, it is automatically included in the request to Claude. Claude shows auth-related files for you to choose from.

**In CLAUDE.md (persistent context):**
```markdown
The database schema is defined in @prisma/schema.prisma.
Reference it when working with data models.
```
When `@` is used inside CLAUDE.md, the referenced file's contents are automatically included in every request. This means Claude can immediately answer questions about the referenced content without reading the file first.

> **Context Window Warning** — Every `@` reference in CLAUDE.md permanently occupies context window space. Only reference files that are genuinely needed across most requests. For files needed occasionally, use interactive `@` mentions in chat instead.

## Instructor Tips

- Run `/init` whenever you start Claude Code in a new project for the first time
- Open and review the generated CLAUDE.md to understand what Claude knows about your project
- The `#` shortcut is faster than manually editing CLAUDE.md — Claude merges instructions intelligently
- Use `@` mentions to save time when you already know which files are relevant
- Put critical cross-cutting files (like database schemas) in CLAUDE.md with `@` so Claude always has that context
- Be selective — too many `@` references in CLAUDE.md wastes context window and degrades performance

## Key Takeaways

1. Too much irrelevant context decreases Claude Code's effectiveness — guide it with just enough
2. `/init` bootstraps project context by analyzing the codebase and generating CLAUDE.md
3. CLAUDE.md is included in every request — it is a persistent system prompt
4. Three CLAUDE.md levels exist: project (shared), local (personal), and global (all projects)
5. `#` memory mode lets you update CLAUDE.md intelligently without manual editing
6. `@` in chat provides one-time context; `@` in CLAUDE.md provides persistent context on every request
7. Referenced files in CLAUDE.md are automatically loaded — Claude can answer instantly without searching

## Familiar Analogies

| Concept | Analogy | Why It Fits |
|---------|---------|-------------|
| CLAUDE.md | `.bashrc` / `.zshrc` — loaded on every shell session | Persistent config that shapes all behavior, loaded automatically |
| CLAUDE.md hierarchy | CSS specificity: inline > class > element | More specific (local) overrides more general (global) |
| `/init` | `git init` + auto-generated README | Bootstraps project with metadata and structure summary |
| `@` in CLAUDE.md | `import` statement at file top | Always loaded, always available in scope |
| Interactive `@` in chat | Dynamic `import()` | Loaded on demand, only when needed for that request |
| `#` memory command | `git config --global` for settings | Persists a directive for all future sessions |
| Context window budget | RAM in a running program | Finite resource — loading too much leaves less room for processing |

## CCA Exam Connection

This lesson covers three high-value exam topics:

**CLAUDE.md hierarchy (Task 3.1)** — Three levels: project (shared via source control), local (personal, gitignored), global (all projects on machine). More local = higher priority.

**Context preservation (Task 5.1)** — CLAUDE.md persists across sessions. `@` references in CLAUDE.md are always loaded. `#` command updates persistent memory.

**Large codebase context (Task 5.4)** — Too much context hurts performance. Use `@` in CLAUDE.md sparingly for cross-cutting files. Let Claude discover the rest via tools.

Key distinctions the exam tests:
- CLAUDE.md (shared) vs CLAUDE.local.md (personal) — both project-level but different sharing scopes
- `@` in CLAUDE.md (every request) vs `@` in chat (one-time) — persistent vs on-demand context
- `/init` (generates initial CLAUDE.md) vs `#` (adds instructions to existing CLAUDE.md)

## Anti-Patterns

| Anti-Pattern | Why It Fails | Correct Approach |
|-------------|-------------|-----------------|
| Adding 15+ `@` references in CLAUDE.md | Consumes context window, degrades performance and accuracy | Keep only cross-cutting files; use interactive `@` for the rest |
| Never running `/init` | Claude starts each session with zero project knowledge | Run `/init` once on first use, then maintain incrementally |
| Using CLAUDE.local.md for team standards | CLAUDE.local.md is gitignored — teammates never see it | Use project CLAUDE.md for shared standards |
| Manually editing CLAUDE.md every time | Tedious and error-prone, easy to break formatting | Use `#` memory mode for intelligent merging |
| Putting personal preferences in project CLAUDE.md | Imposes your style on the whole team via source control | Use CLAUDE.local.md or ~/.claude/CLAUDE.md for personal prefs |
| Ignoring the global ~/.claude/CLAUDE.md | Misses the opportunity for cross-project personal preferences | Set personal coding style in the global file |

## Practice Questions

**Q1.** Your project CLAUDE.md says "use 2-space indentation." Your CLAUDE.local.md says "use 4-space indentation." Your global ~/.claude/CLAUDE.md says "use tabs." Which style does Claude follow?

- A) Tabs (global has highest priority)
- B) 2-space indentation (project CLAUDE.md is the standard)
- C) 4-space indentation (local overrides project and global)
- D) Claude asks you which to use

> **Answer: C.** In the CLAUDE.md hierarchy, more local = higher priority. CLAUDE.local.md overrides project CLAUDE.md, which overrides the global ~/.claude/CLAUDE.md.

**Q2.** A developer adds 12 `@` file references to CLAUDE.md. Responses become slower and less accurate. What is the most likely cause and fix?

- A) Claude Code has a bug with many `@` references — update to latest version
- B) The `@` references consume too much context window — remove non-essential references and use interactive `@` instead
- C) The files are too large — split them into smaller files
- D) CLAUDE.md has a limit of 10 `@` references — remove the extras

> **Answer: B.** The lesson explicitly states that too much irrelevant context decreases performance. Every `@` reference in CLAUDE.md is loaded on every request. The fix is to keep only critical cross-cutting files and use interactive `@` for task-specific files.

**Q3.** Your team wants all developers using Claude Code to follow the same coding standards. Which approach is correct?

- A) Each developer adds standards to their ~/.claude/CLAUDE.md
- B) Add standards to the project CLAUDE.md and commit it to source control
- C) Add standards to CLAUDE.local.md in the project
- D) Use `#` to set standards in each developer's session

> **Answer: B.** Project CLAUDE.md is committed to source control and shared with all team members. This is the architectural approach — no manual setup per developer, no configuration drift.

**Q4.** What is the difference between using `@schema.prisma` in a chat message versus adding `@schema.prisma` to CLAUDE.md?

- A) No difference — both work the same way
- B) In chat, the file is included only in that request; in CLAUDE.md, it is included in every request
- C) In CLAUDE.md, the file is cached; in chat, it is read fresh each time
- D) `@` syntax only works in chat, not in CLAUDE.md

> **Answer: B.** Interactive `@` in chat provides one-time context for the current request. `@` in CLAUDE.md makes the file's contents persistent — automatically included in every request, which trades context window space for instant availability.
