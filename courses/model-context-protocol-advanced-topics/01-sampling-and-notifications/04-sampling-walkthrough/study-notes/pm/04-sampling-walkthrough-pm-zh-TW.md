## 一句話摘要

Sampling 是讓 MCP 伺服器代替自己向客戶端請求 AI 文字生成的機制——把它想成「向總部回報」，外勤人員（伺服器）在任務中向指揮中心（客戶端）請求情報。

---

## PM 為何需要了解 Sampling

Sampling 改變了 MCP 的一般流程。通常是客戶端要求伺服器做事（執行工具、讀取資源）。有了 Sampling，方向反轉：伺服器要求客戶端生成文字。這對產品設計很重要，因為：

1. **伺服器端智慧** — 你的伺服器工具可以在執行過程中融入 AI 推理
2. **客戶端控制 LLM** — 客戶端決定使用哪個模型以及如何呼叫
3. **人機協作潛力** — 客戶端可以在生成文字之前插入人工審核步驟

---

## 心智模型：餐廳廚房

| 角色 | MCP 對應 | 發生的事 |
|------|---------|---------|
| 廚房（伺服器） | MCP 伺服器工具 | 發現需要一個餐酒搭配建議 |
| 服務員跑回去找侍酒師 | Sampling 請求 | 伺服器發送訊息請求文字生成 |
| 侍酒師（客戶端 + LLM） | 客戶端的 sampling 回呼 | 運用專業知識生成紅酒建議 |
| 服務員帶著答案回來 | `CreateMessageResult` | 伺服器收到生成的文字並繼續工作 |

關鍵洞察：廚房（伺服器）永遠不會直接和侍酒師（LLM）對話。它一定是透過服務員（客戶端回呼）。

---

## 六步流程

1. **伺服器需要 AI 文字** — 在工具呼叫期間，伺服器決定它需要 LLM 生成的內容
2. **伺服器發送請求** — 提供描述需求的訊息
3. **客戶端接收請求** — 其預先註冊的回呼函數啟動
4. **客戶端生成文字** — 使用它偏好的任何 AI 模型和 SDK
5. **客戶端回傳結果** — 格式化為標準 MCP 結果
6. **伺服器繼續工作** — 使用生成的文字完成其任務

---

## 產品設計影響

| 考量面向 | 影響 |
|---------|------|
| **模型選擇權在客戶端** | 你的伺服器不需要知道使用哪個 LLM——它只需要請求文字 |
| **成本控制** | 客戶端可以強制限制 token 數量、選擇較便宜的模型或快取回應 |
| **安全性** | API 金鑰保留在客戶端；伺服器永遠看不到 |
| **多步驟工作流程** | 單次工具呼叫可以發起多次 sampling 請求，實現複雜推理鏈 |
| **需要格式轉換** | MCP 訊息格式與 LLM SDK 格式不同——工程團隊必須處理這個轉換 |

---

## 何時在產品中使用 Sampling

| 場景 | 為何用 Sampling | 替代方案 |
|------|----------------|---------|
| 伺服器工具需要摘要檢索的資料 | 伺服器可以在工具執行中請求摘要 | 回傳原始資料讓客戶端摘要（較不優雅） |
| 多代理編排 | 子代理（伺服器）可以向主代理（客戶端）請求推理 | 將所有邏輯建構在一個龐大的客戶端中 |
| 內容生成管線 | 伺服器處理工作流程步驟，透過客戶端委派寫作給 LLM | 硬編碼範本替代 AI 生成 |

---

## 常見誤解

| 誤解 | 事實 |
|------|------|
| 「伺服器直接呼叫 LLM」 | 不是——伺服器向客戶端請求，由客戶端呼叫 LLM |
| 「Sampling 使用與 Anthropic API 相同的訊息格式」 | 不是——MCP 有自己的訊息型別，需要轉換 |
| 「伺服器選擇模型」 | 不是——客戶端的回呼函數決定模型 |
| 「Sampling 是自動的」 | 不是——客戶端必須明確註冊回呼函數 |

## CCA 考試相關性

- D2 經常測試 **sampling 流程方向**（伺服器發起，客戶端執行）
- 預期會有情境題，伺服器工具需要智慧——正確答案涉及 sampling
- 考試測試對客戶端控制模型選擇和 API 金鑰的理解
- 考題可能呈現 sampling 缺失的情境，要求你找出根本原因（缺少回呼註冊）

## 記憶卡

| # | 問題 | 答案 |
|---|------|------|
| 1 | 在 MCP sampling 中，哪個元件發起文字生成請求？ | 伺服器在工具執行期間發起 |
| 2 | 在 sampling 中，哪個元件實際呼叫 LLM？ | 客戶端，透過其 sampling 回呼函數 |
| 3 | 伺服器知道客戶端使用哪個 LLM 模型嗎？ | 不知道——模型選擇完全是客戶端的責任 |
| 4 | 為什麼 sampling 期間需要格式轉換？ | MCP 使用自己的訊息型別，與 LLM SDK 格式（如 Anthropic）不同 |
| 5 | LLM 的 API 金鑰在 sampling 流程中位於哪裡？ | 僅在客戶端——伺服器永遠看不到 |
| 6 | 如果客戶端未註冊 sampling 回呼會怎樣？ | 伺服器的 sampling 請求將會失敗 |
| 7 | 伺服器能否在一次工具執行中發起多次 sampling 呼叫？ | 可以——它可以為複雜的工作流程串連多次請求 |
| 8 | Sampling 如何影響成本控制？ | 客戶端控制 token 限制、模型選擇和快取，擁有完整的成本控制權 |
