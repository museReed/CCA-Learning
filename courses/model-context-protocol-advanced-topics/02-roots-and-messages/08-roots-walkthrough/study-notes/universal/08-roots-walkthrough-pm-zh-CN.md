## 一句话摘要

Roots 是 MCP 的文件系统访问控制机制——用户告诉客户端服务器可以访问哪些目录，而服务器必须在触碰任何文件之前主动检查这些边界。把它想成一个「访客证」系统，限制访客可以进入哪些房间。

---

## PM 为何需要了解 Roots

Roots 直接影响**安全需求**和**用户信任**：

1. **访问控制不是自动的** — MCP SDK 不会自动执行边界。如果工程团队忘记实现检查，服务器可以访问任何文件。
2. **用户定义的权限** — 用户决定分享哪些目录。你的用户体验需要清楚传达他们授予了什么访问权。
3. **合规影响** — 如果你的产品处理敏感数据，root 验证是安全需求，不是「有更好」。

---

## 心智模型：大楼门禁卡

| 大楼安全 | MCP Roots |
|---------|-----------|
| 访客请求一张门禁卡 | 用户通过 CLI 指定允许的目录 |
| 前台制作门禁卡 | 客户端用 `file://` URI 创建 `Root` 对象 |
| 保安在每道门检查门禁卡 | 服务器在文件访问前调用 `is_path_allowed()` |
| 门禁卡只能进特定楼层 | Root 边界限制可访问的目录 |
| **没有保安 = 任何人可以进入** | **没有授权检查 = 服务器可以访问任何东西** |

最后一行是最重要的教训：门禁卡（root）本身什么都不做。没有保安（授权检查），它只是一张塑料卡。

---

## 七步架构

1. **用户指定目录** — 通过 CLI 参数、配置文件或 UI
2. **客户端创建 Root 对象** — 将路径转换为 `file://` URI
3. **客户端注册回调** — 被询问时会返回 roots
4. **服务器请求 roots** — 需要知道允许的路径时回调客户端
5. **服务器实现授权** — 检查每次文件访问是否在 roots 内
6. **服务器将 roots 提供给 LLM** — 通过工具或提示，让 AI 知道有什么可用
7. **服务器执行边界** — 阻止访问 roots 外的路径

## 产品设计影响

| 考量面向 | 影响 |
|---------|------|
| **用户意识** | 用户必须了解他们授予了哪些目录的访问权。清晰的用户体验至关重要。 |
| **默认行为** | 没有 roots 时，服务器没有声明的边界。决定：默认拒绝全部，还是需要明确配置？ |
| **粒度** | Roots 是目录级别的。你无法通过 roots 授予单个文件的访问权。 |
| **动态更新** | Roots 可以在会话期间改变。考虑：你的产品应否支持在会话中添加/移除目录？ |
| **安全由开发者执行** | MCP SDK 不提供内建的执行机制。这必须写在工程需求中。 |

## PM 安全风险矩阵

| 风险 | 无 Roots 时的可能性 | 影响 | 缓解措施 |
|------|-------------------|------|---------|
| 服务器读取敏感文件（`.env`、凭证） | 高 | 重大 | 所有文件读取前强制 root 检查 |
| 服务器写入系统目录 | 中 | 重大 | 所有写入操作都进行 root 验证 |
| 用户不知不觉授予广泛访问权 | 中 | 高 | 清晰的 UX 显示确切分享的路径 |
| 开发者在新工具中忘记授权检查 | 高 | 重大 | 代码审查清单、自动化测试 |

## 验收标准模板

对于任何涉及 MCP 文件访问的功能，你的 PRD 应包含：

- [ ] 服务器在访问前必须验证所有文件路径是否在客户端提供的 roots 内
- [ ] 拒绝访问的错误必须返回清楚说明限制的消息
- [ ] UI 必须显示用户授予了哪些目录的访问权
- [ ] 新工具必须包含 root 验证（加入代码审查清单）
- [ ] 测试用例必须验证 roots 外的路径被拒绝

## CCA 考试相关性

- Roots 跨越 **D2**（MCP 基本要素）和 **D4**（安全/访问控制）——预期跨领域考题
- 最常考的重点：**MCP SDK 不会自动执行 roots**。如果考题描述 MCP 应用的安全漏洞，检查授权是否缺失。
- 回调模式（按需列出 roots）是 D2 概念
- 路径验证作为安全需求是 D4 概念
- 考试哲学：**验证 > 信任**——永远不要假设服务器会在没有明确检查的情况下遵守边界
