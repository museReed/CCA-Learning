## 一句話摘要

Roots 是 MCP 的檔案系統存取控制機制——使用者告訴客戶端伺服器可以存取哪些目錄，而伺服器必須在觸碰任何檔案之前主動檢查這些邊界。把它想成一個「訪客證」系統，限制訪客可以進入哪些房間。

---

## PM 為何需要了解 Roots

Roots 直接影響**安全需求**和**使用者信任**：

1. **存取控制不是自動的** — MCP SDK 不會自動執行邊界。如果工程團隊忘記實作檢查，伺服器可以存取任何檔案。
2. **使用者定義的權限** — 使用者決定分享哪些目錄。你的使用者體驗需要清楚傳達他們授予了什麼存取權。
3. **合規影響** — 如果你的產品處理敏感資料，root 驗證是安全需求，不是「有更好」。

---

## 心智模型：大樓門禁卡

| 大樓安全 | MCP Roots |
|---------|-----------|
| 訪客請求一張門禁卡 | 使用者透過 CLI 指定允許的目錄 |
| 櫃台製作門禁卡 | 客戶端用 `file://` URI 建立 `Root` 物件 |
| 警衛在每道門檢查門禁卡 | 伺服器在檔案存取前呼叫 `is_path_allowed()` |
| 門禁卡只能進特定樓層 | Root 邊界限制可存取的目錄 |
| **沒有警衛 = 任何人可以進入** | **沒有授權檢查 = 伺服器可以存取任何東西** |

最後一行是最重要的教訓：門禁卡（root）本身什麼都不做。沒有警衛（授權檢查），它只是一張塑膠卡。

---

## 七步架構

1. **使用者指定目錄** — 透過 CLI 參數、設定檔或 UI
2. **客戶端建立 Root 物件** — 將路徑轉換為 `file://` URI
3. **客戶端註冊回呼** — 被詢問時會回傳 roots
4. **伺服器請求 roots** — 需要知道允許的路徑時回呼客戶端
5. **伺服器實作授權** — 檢查每次檔案存取是否在 roots 內
6. **伺服器將 roots 提供給 LLM** — 透過工具或提示，讓 AI 知道有什麼可用
7. **伺服器執行邊界** — 阻止存取 roots 外的路徑

---

## 產品設計影響

| 考量面向 | 影響 |
|---------|------|
| **使用者意識** | 使用者必須了解他們授予了哪些目錄的存取權。清晰的使用者體驗至關重要。 |
| **預設行為** | 沒有 roots 時，伺服器沒有宣告的邊界。決定：預設拒絕全部，還是需要明確設定？ |
| **粒度** | Roots 是目錄層級的。你無法透過 roots 授予單一檔案的存取權。 |
| **動態更新** | Roots 可以在會話期間改變。考慮：你的產品應否支援在會話中新增/移除目錄？ |
| **安全由開發者執行** | MCP SDK 不提供內建的執行機制。這必須寫在工程需求中。 |

---

## PM 安全風險矩陣

| 風險 | 無 Roots 時的可能性 | 影響 | 緩解措施 |
|------|-------------------|------|---------|
| 伺服器讀取敏感檔案（`.env`、憑證） | 高 | 重大 | 所有檔案讀取前強制 root 檢查 |
| 伺服器寫入系統目錄 | 中 | 重大 | 所有寫入操作都進行 root 驗證 |
| 使用者不知不覺授予廣泛存取權 | 中 | 高 | 清晰的 UX 顯示確切分享的路徑 |
| 開發者在新工具中忘記授權檢查 | 高 | 重大 | 程式碼審查清單、自動化測試 |

---

## 驗收標準範本

對於任何涉及 MCP 檔案存取的功能，你的 PRD 應包含：

- [ ] 伺服器在存取前必須驗證所有檔案路徑是否在客戶端提供的 roots 內
- [ ] 拒絕存取的錯誤必須回傳清楚說明限制的訊息
- [ ] UI 必須顯示使用者授予了哪些目錄的存取權
- [ ] 新工具必須包含 root 驗證（加入程式碼審查清單）
- [ ] 測試案例必須驗證 roots 外的路徑被拒絕

---

## CCA 考試相關性

- Roots 跨越 **D2**（MCP 基本要素）和 **D4**（安全/存取控制）——預期跨領域考題
- 最常考的重點：**MCP SDK 不會自動執行 roots**。如果考題描述 MCP 應用的安全漏洞，檢查授權是否缺失。
- 回呼模式（按需列出 roots）是 D2 概念
- 路徑驗證作為安全需求是 D4 概念
- 考試哲學：**驗證 > 信任**——永遠不要假設伺服器會在沒有明確檢查的情況下遵守邊界
