## 一句話總結

`tool_choice` 是產品層面的控制桿，決定了 Claude 在選擇下一步動作時擁有多少自由度 — 用它在「對話彈性」與「工作流可靠度」之間做取捨。

---

## 心智模型 — 在餐廳點餐

把 `tool_choice` 想像成在餐廳怎麼點餐：

| 模式 | 餐廳類比 | 產品意義 |
|------|----------|----------|
| `auto` | 「給我一個驚喜 — 聊聊天也好，吃東西也好」 | Claude 自行判斷這一輪要呼叫工具還是純聊天 |
| `any` | 「我一定要從你菜單點菜，但你幫我挑」 | 每一輪都必須產出工具呼叫，由 Claude 挑選 |
| `tool` | 「我就是要千層麵，不要其他的」 | 強制呼叫特定工具 — 保證結構化輸出 |
| `none` | 「告訴我今天有什麼好東西就好，不點餐」 | Claude 只會用純文字回應，本輪工具關閉 |

關鍵張力在於：`auto` 的使用體驗最自然，但最不可預測；`tool` 最可預測，但會殺掉對話感。你的產品在不同時機都會需要這兩種模式。

---

## 產品使用情境

### 何時使用 `auto` — 對話式 Agent

- 客服聊天機器人 — 有時需要查訂單，有時只是純聊天。
- 寫程式助手 — 一般問題直接回答，只在使用者要求檔案操作時才呼叫工具。
- **理由**：使用者期待自然對話。對「謝謝！」強制呼叫工具會讓人覺得產品壞掉。

### 何時使用 `any` — 動作導向的工作流

- Agentic 任務執行器（「幫我訂機票、再訂飯店、再訂車」），每一輪都必須推進計畫。
- 後台自動化場景 — 人類閒聊不在範圍內，每一輪 Claude 都應該發 API。
- **理由**：排除 Claude 在工作流中途「解釋」而不是「行動」的風險。

### 何時使用 `tool` — 結構化資料抽取

- 履歷解析、發票 OCR、從非結構化 email 填表、分類到標籤體系。
- 任何你過去會用「JSON mode」或嚴格輸出 schema 的場景。
- **理由**：該工具的 `input_schema` 變成一份有型別的輸出契約。不會有 prompt 脆弱性，也不用靠「請用 JSON 回覆」這種黑魔法。

### 何時使用 `none` — 總結與反思輪

- 多步驟 agent 收集完所有資料後，最後要輸出給使用者看的那一輪用 `none`，防止 Claude 又觸發一次工具呼叫。
- 評估框架中，你要 Claude 為某個回應打分但不要它呼叫任何東西。
- **理由**：給你的 agent 一個明確的「停下來講話」訊號。

## PM 決策框架

依序問自己這四個問題：

1. **這一輪有可能純粹是對話嗎？**
   - 是 → 從 `auto` 開始。
   - 否 → 往下。
2. **這一輪需要產出恰好一種命名的輸出格式嗎？**
   - 是 → `tool` 並指定工具名稱。
   - 否 → 往下。
3. **這一輪需要從動作菜單中挑一個，但一定要做一個？**
   - 是 → `any`。
   - 否 → 往下。
4. **這一輪是反思、總結、或最終要給使用者看的回覆嗎？**
   - 是 → `none`。

在 agent 產品中，同一個 loop 的不同輪次會使用不同模式。常見 pattern：從 `auto` 開始，進入執行階段時切到 `any`，最後總結時切到 `none`。

---

## 常見 PM 錯誤

1. **使用者期待對話時卻強制 `any`** — 你的聊天機器人開始對「你好」回覆奇怪的工具呼叫，使用者會以為產品壞了。
2. **忘了 `tool` 和 `any` 會殺掉推理透明度** — 當 Claude 被強制時，它不會在動作前「想出聲」。你會失去 `auto` 模式免費提供的可解釋性。這對受監管產業（金融、醫療）很重要 — 稽核軌跡必須展示某個動作「為什麼」被採取。
3. **用 `auto` 做結構化資料抽取** — Claude 可能用聊天文字回覆（「好的！以下是你要的資料⋯⋯」），而不是乾淨的工具呼叫，下游 parser 就爆了。應該用 `tool`。
4. **把 `tool_choice` 當成全域設定** — 它是每次 API 呼叫各自設定的。聰明的 agent 產品會逐輪切換模式，根據當前 agent 處於什麼階段。
5. **後端無法並行時沒設 `disable_parallel_tool_use`** — Claude 可能同時發出兩個工具呼叫，你的單執行緒執行器會出狀況。後端不支援並行時，明確強制序列執行。

---

> **關鍵洞察**
>
> `tool_choice` 是「使用者體驗」與「產品可靠度」之間的旋鈕。`auto` 最大化體驗（自然對話、透明推理）但犧牲保證；`any` 和 `tool` 最大化可靠度（保證結構化動作）但犧牲對話感與推理透明度。最好的 agent 產品不會只選一種 — 它們會**在同一個使用者 session 內**切換模式：用 `auto` 聊天、用 `any` 執行、用 `none` 總結。

---

## CCA 考試相關性

- **D2（Tool Design & MCP Integration）**：PM 風格的題目可能會問「哪個設定能保證模型回傳結構化資料？」— 答案是 `{"type": "tool", "name": "..."}`。
- **D1（Agentic Architecture）**：預期會看到情境題問 agent loop 中應該如何在不同輪次切換 `tool_choice`。
- **陷阱題**：考題可能描述一個「有時需要查資料」的聊天助手，正確答案是 `auto`，不是 `any`。

## 快閃卡

| 正面 | 背面 |
|------|------|
| 哪個 `tool_choice` 模式最像正常的對話？ | `auto` — Claude 自行決定要聊天還是呼叫工具，閒聊還能自然運作。 |
| 如果你的工作流每一輪都必須產出結構化動作，應該選哪個模式？ | `any` — Claude 必須呼叫所提供工具中的一個，由它自行挑選。 |
| 結構化資料抽取（「JSON mode」情境）最適合用哪個模式？ | 指定名稱的 `tool` — 工具的 schema 就是有型別的輸出契約。 |
| `none` 的作用是什麼？什麼時候有用？ | 停用本輪的工具呼叫；適合 agent loop 最後的總結／反思輪。 |
| 使用 `any` 或 `tool` 時，你會失去什麼推理行為？ | 工具呼叫前的 chain-of-thought 文字 — Claude 會直接輸出 tool_use block，可解釋性就沒了。 |
| 為什麼 `any` 不適合當作客服聊天機器人的預設值？ | 它會在每一輪都強制工具呼叫，連「你好」或「謝謝」都會觸發毫無意義的呼叫。 |
| 設計良好的 agent 產品什麼時候會在 session 中途切換 `tool_choice`？ | 聊天用 `auto`、執行階段用 `any`、最後總結用 `none` — 不同階段用不同模式。 |
| `disable_parallel_tool_use` 緩解了什麼產品風險？ | 防止 Claude 在後端無法安全並行執行時同時觸發多個動作。 |
| 法務團隊需要稽核 Claude 為什麼採取某個動作，哪個模式能保留推理？ | `auto` — 只有 `auto` 允許 Claude 在工具呼叫前輸出自然的 chain-of-thought 文字。 |
| `tool_choice: none` 代表你應該從 request 中省略 `tools=` 嗎？ | 不是 — 工具列表還是要宣告；`none` 只是停用本輪的呼叫行為。 |
