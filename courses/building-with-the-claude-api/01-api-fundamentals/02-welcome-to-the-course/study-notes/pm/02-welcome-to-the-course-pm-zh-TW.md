## 產品能力地圖

把每個課程模組想成你的產品團隊可以解鎖的一項能力：

| 模組 | 為產品帶來的能力 | PM 決策 |
|------|----------------|---------|
| API 基礎 | 基本 AI 文字生成 | 「我們能不能呼叫 Claude？」——基本門檻 |
| 提示評估 | AI 輸出的品質保證 | 「我們怎麼知道 AI 在正常運作？」——最重要的模組 |
| 提示工程 | 控制 AI 語氣、格式、準確度 | 「我們怎麼讓 AI 做我們想要的事？」 |
| 工具使用 | Claude 呼叫外部函式 | 「Claude 能查資料、訂機票嗎？」 |
| RAG | 讓 AI 基於公司文件回答問題 | 「Claude 能回答關於我們產品的問題嗎？」 |
| MCP | 與外部服務的標準化整合 | 「我們怎麼把 Claude 接到 Slack、資料庫等？」 |
| Claude Code / Computer Use | AI 自主操作軟體 | 「AI 能替我們寫程式碼或瀏覽網頁嗎？」 |
| 工作流與代理 | 多步驟 AI 流程 | 「我們能自動化整個工作流，而不只是單一查詢嗎？」 |

## PM 為什麼該在意這份路線圖

課程遵循明確的相依鏈。這對產品規劃至關重要，因為：

1. **沒有工具使用就無法交付代理功能** —— 如果你的路線圖把「AI 代理」放在 Q1、「工具整合」放在 Q2，這個計畫就是壞的。
2. **提示評估是品質的把關者** —— 講師稱之為最重要的實踐。跳過評估規劃的 PM 會得到 demo 很好看、但生產環境失敗的 AI 功能。
3. **RAG 解鎖企業價值** —— 大多數企業客戶都要求 AI 了解他們的資料。RAG 就是實現這件事的地方。

## PM 決策框架

| 要問團隊的問題 | 為什麼重要 |
|--------------|-----------|
| 我們的 API 存取和金鑰設好了嗎？ | 卡在這裡 = 工程速度歸零 |
| 我們在上線前有規劃提示評估嗎？ | 沒有評估 = 沒有品質保證；講師最強調這一點 |
| 哪個模組對應我們的下一個功能？ | 優先學習路線圖最需要的模組 |
| 我們在建構工具還是代理？ | 工具是確定性的、可測試的；代理更靈活但更難評估 |
| 我們的使用場景需要 RAG 嗎？ | 如果使用者會問關於專有資料的問題，答案就是需要 |

## 常見 PM 錯誤

1. **在不了解先決條件的情況下規劃 AI 代理** —— 代理需要提示工程 + 工具使用 + 評估。跳過任何一項都會導致不可靠的功能。
2. **把提示工程當成「工程師的事」** —— 提示設計決策就是產品決策（語氣、格式、準確度目標）。
3. **忽略提示評估** —— 講師明確說過：不做評估就部署，你的使用者不會得到你預期的結果。
4. **一次規劃太多能力** —— 相依鏈意味著你應該先交付 API 基礎、再交付工具、再交付代理。
5. **沒有為迭代留出時間** —— 課程強調擴展、修改和破壞 Notebook。你的衝刺計畫需要留出提示迭代的空間。

> **核心洞察**
>
> 這份路線圖不只是課程大綱——它是 AI 功能的產品成熟度模型。公司會按這些階段推進：基本 API 呼叫、然後評估、提示工程、工具使用、RAG、MCP，最後才是代理。試圖跳過階段，就是 AI 功能在生產環境失敗的原因。
